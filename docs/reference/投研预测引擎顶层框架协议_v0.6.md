# AI Agent 投研预测引擎顶层框架协议 v0.6

> 目标：一个人机交互下的、数据驱动的、持续运行的财务预测引擎。  
> 当前正式产出：契约口径下的公司未来季度 / 年度净利润预测。  
> 未来演进方向：从净利润预测，逐步扩展到利润表、资产负债表、现金流量表的完整预测。

---

## 版本说明

v0.6 基于 v0.5 继续收口，核心调整是重新切分 L2 / L3。

v0.5 的主要问题不是方向错误，而是 L2 / L3 的职责边界还不够干净：

- L2 被命名为“入库处理带”，更像一个处理流程；
- L3 被命名为“证据账本”，同时承担原始 claims、当前采信视图、关系型 claims、derived claims 和冲突状态；
- 这样在真实运行时会导致一个问题：上游资料变化、人工裁决变化、口径契约升版、框架条目变化时，哪些对象该失效、哪些对象该重算，边界不够清楚。

v0.6 的核心调整是：

1. **L2 改为“观测证据层”**。它不再只是入库流程，而是长期维护 source-backed claims：来源直接支持的原始 claims、当前采信视图、冲突状态、关系型 claims、人工裁决、冷存 / 弱信号触发。L1 → L2 的“入库处理带”只是 L2 的入口动作。
2. **L3 改为“派生证据层”**。它专门管理 derived claims：由 L2 观测 claims 经过口径契约、对账规则、计算规则、框架映射、聚合逻辑生成的派生事实、归一化事实、对账后事实和跨公司 / 行业 / 主题 / 宏观视图。
3. **公司、行业、宏观、主题不是单独层，而是 L2 / L3 的主体域和作用域**。同一条信息可能从主题影响行业，再影响公司预测，不能硬拆成四套架构。
4. **L5 优先消费 L3，但必须能回溯 L2**。预测层通常使用 L3 的对账后、归一化、派生后的输入；但正式快照必须能追到 L2 的原始来源 claims。
5. **L6 的血缘边同步调整**：`source_ref / document_unit → observed claim → current evidence view → derived claim → assumption → forecast_node → snapshot`。

这版文档仍然不写代码、不写 SQL、不写详细 schema、不写目录结构。它只锁定顶层架构、层间职责、运行机制和硬边界。

# 0. 系统宪法

## 0.1 系统本质

这不是问答系统，不是单次 case 驱动的研究助手，也不是“人提问、机器搜索、返回答案”的工具。

它是：

> **一个人机交互下的、数据驱动的、持续运行的财务预测引擎。**

系统长期吸收资料，沉淀证据，维护公司预测模型，发现冲突，触发研究课题，更新假设，生成预测快照，并在财报季复盘自己的错误。

## 0.2 唯一正式产出

系统唯一正式产出是：

> **公司未来财务预测。**

当前阶段对外输出：

> **契约口径下的未来季度 / 年度净利润预测。**

“契约口径”可以很轻：

- v0：原始财报口径；
- v0.1：原始财报口径 + 扣非；
- 后续：逐步加入公司特定调整项、一次性项目、会计政策变化处理、业务重组口径等。

## 0.3 系统不做什么

系统不做：

- 估值；
- 选股；
- 买卖建议；
- 判断便宜与否；
- 组合构建；
- 市场短期博弈判断。

这些由投资经理拿着预测结果另行判断，不进入本系统边界。

## 0.4 设计原则

1. **预测优先**：任何模块存在的理由，都必须能解释它如何改善财务预测，或改善预测的可解释性、可复盘性。
2. **证据优先**：系统不直接相信观点。观点必须被拆成 claim，进入观测证据层，被标注来源、口径、时间、置信度和冲突状态。
3. **派生有边界**：derived claims 必须有输入 claims、规则版本、口径契约版本、框架引用和 as-of 语义，不能变成第二套独立真相。
4. **披露为锚**：财报、公告、业绩预告、问询函回复是硬锚；外部细分数据可以补充披露锚，但不能无解释地违反披露锚。
5. **L1 可轻，L2 / L3 必须重**：原始资料层不强求全量 raw 归档，但观测证据、派生证据、假设账本、快照和日志必须可解释。
6. **推拉双向**：既有数据流入后自动传播的“推”，也有从预测问题出发主动找资料的“拉”。
7. **课题不是旁支**：研究课题是调度行动的一类，不是第二套系统。课题拿到的资料仍然必须走 L1 登记、L2 观测证据、L3 派生证据，再进入 L5。
8. **规则与调度分离**：规则是判断依据，调度是运行控制，行动是一次执行。调度不承载研究方法本身。
9. **人批关键认知变化**：框架、口径契约、重大冲突裁决、正式预测发布必须有人参与。
10. **LLM 不做确定性工作**：能用确定性规则做的，不交给 LLM。LLM 主要负责抽取、归纳、消歧、提出候选解释和生成待办。
11. **快照不可回改**：预测快照发布后不可修改，只能产生新快照或重述复盘。

# 1. 总体形态

系统由一条主管道、一个旁路层、一根调度脊柱、一个存储与运维底座、两条记录线和一个拉式课题回路组成。

```text
                        ┌────────────────────────┐
                        │  L4 知识与规则层         │
                        │  框架库 / 规则工件库     │
                        └───────────┬────────────┘
                                    │ 加载规则 / 引用框架
                                    ▼
┌──────────────┐    ┌────────────────┐    ┌────────────────┐    ┌──────────────┐    ┌──────────────┐
│ L1 资料资产层 │ -> │ L2 观测证据层   │ -> │ L3 派生证据层   │ -> │ L5 公司预测层 │ -> │ 预测快照      │
│ 来源/载体/追溯│    │ source-backed  │    │ derived claims │    │ 驱动树/假设   │    │ 正式/影子     │
└──────────────┘    │ claims/current │    │ 归一/对账/聚合 │    └──────────────┘    └──────────────┘
        ▲           └────────────────┘    └────────────────┘            │
        │                    ▲                   ▲                      │
        │                    └────── 课题主动获取资料 ───────────────────┘
        │
        └──────── web / API / MCP / 券商库 / 官网 / 人工上传 / 公告PDF服务

               ┌──────────────────────────────────────────────┐
               │ L6 调度脊柱：感知 / 触发 / 准入 / 行动 / 队列 │
               └──────────────────────────────────────────────┘

               ┌────────────────────┐      ┌────────────────────┐
               │ 执行日志 M-LOG      │      │ 人机待办            │
               │ 输入/输出/版本/动作 │      │ 裁决/调研/补证据     │
               └────────────────────┘      └────────────────────┘

               ┌──────────────────────────────────────────────────┐
               │ 存储与运维底座（横切）：文件存储 / DB / 备份策略 │
               └──────────────────────────────────────────────────┘
```

## 1.1 主管道：数据推动预测

主管道是：

```text
资料接入与披露锚归档
→ 观测证据层
→ 派生证据层
→ 公司预测层
→ 预测快照
```

当新公告、新财报、新纪要、新研报、新 API 数据、新网页资料进入时，系统不是直接改预测，而是先进入 L1 形成可追溯来源资产，再在 L2 形成 source-backed claims，然后在 L3 形成需要的归一化、对账后、派生后的证据，最后由 L6 判断是否影响 L5 的假设和预测。

## 1.2 L1 / L2 / L3 的本质关系

L1、L2、L3 都属于数据处理链，只是精细度不同。

```text
L1 = 资料与来源资产：数据从哪里来，能否追溯。
L2 = 观测证据：来源直接说了什么，哪些被采信，哪些冲突。
L3 = 派生证据：系统基于证据和规则构造出了什么。
```

其中，L2 不是一次性 pipeline，L3 也不是普通缓存。二者都是长期资产：

- L2 长期保存 source-backed claims、当前采信视图、冲突与人工裁决；
- L3 长期保存 derived claims、口径重构值、对账后事实、聚合视图和其输入血缘。

## 1.3 主体域：公司 / 行业 / 宏观 / 主题

公司、行业、宏观、主题不是单独层，而是 L2 / L3 的主体域和作用域。

同一条信息可能天然跨域，例如：

```text
GB300 出货节奏变化
→ 主题观测 claim
→ 影响 AI 服务器产业链
→ 影响光模块行业需求
→ 影响中际旭创 / 新易盛 / 沪电股份等公司
→ 影响收入、毛利率、资本开支假设
```

因此系统不建立“四套 L3”。更合理的是：

```text
L2 支持 company / industry / macro / theme 四类观测 claims；
L3 支持 company / industry / macro / theme 四类 derived claims；
L6 负责跨主体域传播影响；
L5 只接收最终会影响具体公司预测的输入。
```

## 1.4 拉式回路：预测问题反向找资料

系统不能只等待数据流入。新增个股、模型冷启动、关键假设缺失、冲突无法裁决时，必须从预测问题反向发起研究课题：

```text
预测变量 / 阻塞问题
→ 研究课题
→ 查 L2 观测证据和冷存池
→ 查 L3 派生证据
→ 主动获取资料
→ L1 登记来源
→ L2 形成 / 更新观测 claims
→ L3 形成 / 更新派生 claims
→ 更新假设 / 驱动树 / 框架条目
→ 重算预测
```

例如，中际旭创某类产品毛利率不清楚，不是让 agent 直接回答“毛利率是多少”，而是立一个绑定目标变量的课题：

```text
目标变量：中际旭创 800G 产品线 2026Q2 / 2026E 毛利率假设
阻塞问题：该毛利率当前只有一个托底标量，对净利润敏感度高、置信度低
课题类型：结构课题
信息需求：ASP、客户结构、竞争格局、原材料、良率、工艺、产能利用率
处理路径：先定 L4 毛利拆解模板与口径 → 改 L5 驱动树骨架 → 登记数据绑定 → 补 L2 观测证据 → 形成 L3 毛利派生区间 / 基准值 / 冲突标记 → 更新假设 → 重算 → 生成影子预测
```

课题不是第二套系统，它只是 L6 行动库里的一类复合行动。

## 1.5 旁路层：知识与规则

知识与规则层不直接装事实。它装两类东西：

1. **框架库**：怎么理解一个行业、公司、业务、技术路线、商业模式、财务结构。
2. **规则工件库**：怎么抽取、清洗、对账、仲裁、建模、复盘、设定口径契约。

它被 L2、L3、L5、L6 和课题回路加载使用，但不替代证据层。

## 1.6 调度脊柱

调度脊柱负责五件事：

```text
感知 → 触发 → 准入 → 行动 → 队列
```

- 感知：知道来源、观测 claims、派生 claims、假设、框架、契约、快照之间的大致依赖关系。
- 触发：识别新资料、新 claim、状态变化、人工指令、财报披露、模型陈旧等事件。
- 准入：判断是否值得执行、立即执行、低优先入队、仅记录或转人工。
- 行动：执行抽取、校验、对账、检索、派生、重算、生成待办、课题推进等动作。
- 队列：管理失效、重算、低优先任务、人工待办和失败重试。

## 1.7 存储与运维底座（横切）

存储与运维不是某一层的私产，而是横切所有层的**底座**，因此单列在此：

- **文件存储**：自维护披露锚的原始字节（公告 / 财报 PDF、人工上传原件）与派生工件（parser artifact、normalized IR 等），按路径 + 哈希存放。
- **数据库**：company / security、`source_ref`、document、processing_run、document_unit、observed claim、derived claim、假设、快照、日志——同一物理库内按职责分 schema，不按层分库。
- **运维**：schema 迁移、角色与权限、备份、保留与清理策略、容量与可用性。

它服务 L1–L6 的全部职责，但**不属于其中任何一层**。各层只负责自己的动作并把产物写进底座；没有哪一层要去“当数据库 / 文件系统的管理员”。职责分层，存储分 schema，两者正交。

# 2. L1：资料资产层（来源接入与可追溯数据资产）

## 2.1 L1 的定位

L1 是系统把外部资料变成**可引用、可复查、可进入观测证据层的数据资产**的逻辑层。

所有进入系统的数据——标准结构化数据、披露文件、人工资料、轻载外部来源——都先在 L1 完成来源、取数、登记、追溯、可用化。不同来源的数据形态不同，不要求塞进同一张表；但最终都要能通过统一的来源引用进入 L2。

`disclosure_anchor` 只是 L1 里负责披露文件侧的一条路径，不等于整个 L1。

**L1 负责回答：**

- 数据来自哪里；
- 通过什么 adapter / provider / 工具取得；
- 什么时候取得；
- 查询参数、版本、口径是什么；
- 是否需要自维护原文或快照；
- 下游通过什么契约消费它。

**L1 不负责：**

- 抽取 claim；
- 判断投资重要性；
- 判断冲突；
- 形成采信视图；
- 生成 derived claims；
- 改预测。

实现上 L1 是一组**按来源划分、各自带 adapter 的接入服务**。不同来源留下不同形态的数据资产，见 §2.2。

## 2.2 L1 的数据资产分型

L1 的本质不是“存不存文件”，而是：每类来源进入系统后**留下什么可追溯的数据资产**。

| 类型 | 例子 | L1 留什么 | 下游怎么用 |
|---|---|---|---|
| **标准结构化数据** | Wind / iFinD / Choice / Tushare | `dataset_key`、`provider_adapter`、查询参数、返回记录、`source_ref` | `Dataset API` → L2 |
| **自维护披露锚** | 财报、公告、问询函回复 | 原文、hash、`document` / `document_unit`、parser artifact | `Filing API` → L2 |
| **人工 / 自有资料** | 调研纪要、路演笔记、专家访谈 | 资料类型、公司 / 行业关联、来源、时间、可复核性、敏感标记，可选原件 | 作为自有来源进入 L2 证据链 |
| **轻载外部来源** | web、MCP、新闻、论坛、论文 API | `source_ref`、访问时间、工具、查询参数、定位线索 | 只登记来源，不强制转 `document_unit` |

按留存策略，四型归两类：

- **自维护载体**：披露锚、人工资料；
- **只登记来源**：标准数据、轻载外部。

## 2.3 自维护载体策略：哪些来源需要保存原文

这两类来源由 L1 自己留载体：

1. 自维护披露锚；
2. 人工 / 自有资料。

原文 / 快照落入 §1.7 存储底座；解析与 A 类清洗在各服务内完成，产出统一 `document_unit`。

### 2.3.1 上市公司公告 / 财报 PDF

最重要的自维护资产是披露锚：

- 年报；
- 半年报；
- 季报；
- 业绩预告；
- 业绩快报；
- 正式公告；
- 问询函及回复；
- 交易所监管函件；
- 重大资产重组、股权激励、定增、回购、分红等专项公告。

每份原则上存四组信息：

| 组 | 存什么 |
|---|---|
| 元数据 | 来源渠道、下载时间、公告日期、标题、公司代码、公告编号 |
| 原文 | 原始 PDF、文件哈希 |
| 抽取产物 | 文本 / 表格抽取版本；`document_unit` 快照；派生工件（parser artifact / normalized IR / B 类视图，版本化、指回 unit、可被取代、与锚分开存） |
| 定位（可选） | 页码 / 段落 / 表格坐标，供人工复核；G0 身份不依赖坐标 |

存这些是为了能回答：某个净利润预测里的某个假设，最底层锚在哪份公告的哪个 `document_unit`、来源文件哈希是什么、是哪个披露口径。

### 2.3.2 人工上传资料

包括：

- 路演笔记；
- 调研纪要；
- 专家访谈；
- 微信群关键归档；
- 公司跟踪笔记；
- PPT / Excel / 图片 / 录音转写。

这类多是已转写、整理后才上传，所以重点不在“是否留有原件”，而在**能否准确归类关联到公司 / 行业 / 主题，日后能否对账**。

存三组信息：

| 组 | 字段 |
|---|---|
| **必填 · 结构** | 资料类型；涉及公司（名 + 代码，可多个）；行业 / 板块 / 产业链环节；主题标签；资料对应时间（事件 / 会议时间，非上传时间）；上传时间 |
| **轻量 · 可信度** | 来源 / 谁提供；是否一手 vs 转述；是否可复核（能否拿财报 / 公告对账，非是否留原件）；对应 Tier F 与追溯等级 |
| **可选 · 标记** | 是否附原始文件；是否涉敏感 / 未公开重大信息 |

### 2.3.3 载体加工：解析 + A 类清洗 → document_unit

自维护载体在各接入服务内完成解析与清洗，产出 `document_unit`。

解析与 A 类清洗现阶段**内置在各服务里，不抽共享组件**。原因是来源格式各异，本就定制。但各服务必须**产出统一形态的 `document_unit`**，这样 L2 才能跨来源消费。

`document_unit` 是最小可追溯单元，可以是：

- 文本段；
- 表格；
- Q&A；
- 图片说明；
- 转写片段。

身份锚定靠**文件哈希 + 不可变 unit 快照**，并记录 normalization 的 method / version 以便复现；page / bbox 只是可选复核信息。

L1 只做确定性清洗：

| | **A 类 · 在 L1** | **B 类 · 在 L2** |
|---|---|---|
| 性质 | 稳定可枚举规则，不做语义判断 | 需上下文判断，标准随时间演进 |
| 例子 | 页眉页脚、页码、水印、控制字符、空行、明显法律声明、释义、目录、签署页 | 长文摘要、按信息价值取舍、语义精简、复杂内容重组 |
| 形态 | 直接清除；raw 按字节保真留存、可还原 | 版本化派生视图、指回不可变 unit |

**安全红线：** 整段清除的前提是“规则稳定”且“该板块基本无实质信息”。像“重要提示”这种常含退市风险、业绩大变的板块，即便形似套话也不得按标签整段删。

## 2.4 轻载来源策略：哪些来源只登记，不自维护全文

这类来源当前不强求自维护全量 raw：

- 购买的本地数据服务；
- 券商数据库 API；
- Wind / iFinD / Choice 等结构化数据；
- MCP 工具返回；
- web search；
- 公司官网；
- 政府 / 协会 / 海关 / 行业数据库网页；
- 论文 API；
- 新闻站点；
- 自媒体 / 大 V / 论坛。

它们只登记来源。这条访问记录就是 L1 的 `source_ref`。

一条 `source_ref` 至少记：

| 字段 | 说明 |
|---|---|
| 来源标识 | URL / 结果 ID / dataset key / provider ID |
| 访问时间 | 记录 as-of |
| 访问工具 + 查询参数 | 决定是否可重查 |
| 追溯等级 | G0–G4 |
| 支撑定位 | 原文片段 / 页码 / chunk ID / 关键词 / 偏移量之一；会进预测的才必填 |

由此有三条推论：

- **查空 / 检索未得** = 一条没有 claim 的 `source_ref`，也要记录；
- **URL 以后失效** → `source_ref` 仍在，按追溯等级降级，不当作没记过；
- **只有自维护原文需要落盘** → 公告 / 财报 PDF、人工上传件是字节，写入文件存储并由 `source_ref` 记路径 + 哈希。

L1 不追求“我拥有一切原文”，但必须记录“我为什么认为这条证据曾经存在、来自哪里、什么时候看到、以什么形式进入系统”。

## 2.5 来源分级

来源分级不是为了机械打分，而是为了决定：能否采信、能否单独影响预测、是否需要对账、是否只能进弱信号池。

| 层级 | 类型 | 例子 | 使用原则 |
|---|---|---|---|
| Tier 0A | 硬披露锚 | 财报、公告、业绩预告、业绩快报、问询函回复 | 最高优先级；原文必存；用于对账和复盘 |
| Tier 0B | 软披露锚 | 投关活动记录、业绩说明会、官网材料、公司公开路演 | 可提升置信度；仍需结合硬披露锚 |
| Tier 1 | 结构化外部数据 | 券商库、政府统计、协会、海关、产业数据库 | 可作为重要输入；需记录口径和查询时间 |
| Tier 2 | 外部研究与观点 | 研报、纪要、专家访谈、权威媒体深度报道 | 可形成估算和观点型 claims；需对账、冲突检测 |
| Tier 3 | 弱公开信源 | 自媒体、评论员、大 V、论坛、社群传闻 | 默认进弱信号池；不能单独成为正式预测采信依据 |
| Tier F | 自有一手资料 | 自己调研、专家交流、路演笔记 | 高价值但不天然高置信；看可复核性和披露锚一致性 |

## 2.6 追溯等级

因为 L1 不追求全量 raw，自然要接受不同追溯等级。

| 追溯等级 | 含义 | 例子 | 对置信度的影响 |
|---|---|---|---|
| G0 | 原文级追溯 | 原始文件 + 文件哈希 + 不可变 `document_unit` 快照 + 来源链；page / bbox 仅为可选复核信息 | 最高；可作为硬锚 |
| G1 | 数据服务级追溯 | 数据源 + 查询参数 + 返回字段 + 时间 | 可正式使用，但要注意供应商口径 |
| G2 | 网页 / 工具级追溯 | URL / 工具名 + 访问时间 + 关键片段 | 可辅助使用；重要结论需交叉验证 |
| G3 | 人工记录级追溯 | 会议笔记、访谈纪要、转述记录 | 可用但需标记不确定性 |
| G4 | 弱追溯 | 来源模糊、截图不完整、二手传闻 | 原则上只进冷存或待核验，不直接影响正式预测 |

硬规则：

- 追溯等级必须进入 L2 claim 元数据；
- 弱追溯 claim 必须有置信度上限；
- G4 不能单独支撑正式预测；
- 如果弱追溯信息很重要，应触发课题或人工待办，而不是直接采信。

## 2.7 检索记录也是数据

主动 search、API 查询、MCP 调用即使没有找到结果，也要记录。

“查过但没找到”有三个价值：

1. 避免重复检索；
2. 说明某个结论目前缺证据，而不是系统忘了查；
3. 可转为人机待办：例如“公开渠道未找到某产品毛利率，需要问卖方 / 专家 / 公司”。

# 3. L2：观测证据层（Observed Evidence Layer）

## 3.1 L2 的定位

L2 是系统把 L1 的资料资产转成**来源直接支持的证据 claims**的层。

它不应再被理解为一次性“入库处理带”。入库处理只是 L2 的入口动作。L2 的长期职责是维护：

- source-backed 原始 claims；
- 当前采信 claims 视图；
- contested / rejected / provisional / scenario-set 等状态；
- 关系型 claims；
- 冷存 / 弱信号；
- 人工裁决记录；
- as-of 查询所需的历史采信视图。

L2 负责回答：

> **来源直接说了什么？这些说法来自哪里？现在是否被采信？是否有冲突？**

L2 不负责回答：

> **系统基于这些说法推导出了什么？**

后者属于 L3。

## 3.2 L2 的输入

L2 接收两类输入：

1. **来自 L1 自维护载体的 `document_unit`**：公告、财报、问询函、人工纪要、转写文本等；
2. **来自 L1 轻载来源的 `source_ref` + 返回记录 / 摘要 / 定位线索**：API、MCP、web、数据库查询、新闻等。

L2 不关心来源是 PDF、网页、API 还是人工笔记。它关心的是：这些输入能否拆成可判断、可追溯、可冲突、可采信的 claims。

## 3.3 L2 的主体域

L2 claims 支持四类主体域：

| 主体域 | 例子 | 说明 |
|---|---|---|
| company | 公司、子公司、业务线、产品、客户、供应商 | 最终通常会进入公司预测 |
| industry | 行业、产业链环节、技术路线、供需格局 | 常作为行业驱动或横向比较输入 |
| macro | 政策、利率、汇率、关税、贸易限制、宏观需求 | 只有影响公司 driver 时才进入 L5 |
| theme | GB300、CPO、先进封装、国产替代、AI 服务器等 | 主题本身不是投资结论，只是跨公司影响路径 |

主体域不是四套存储，也不是四个系统。它只是 claim 的组织方式和传播边界。

## 3.4 L2 的标准处理链

L2 的标准处理链是：

```text
接收 L1 document_unit / source_ref
→ 轻处理（确定性语义预处理）
→ B 类语义清洗（可选，生成版本化派生视图，指回不可变 unit）
→ 主体 / 时间 / 指标 / 事件识别
→ claim 抽取
→ 口径标注
→ 来源与追溯等级标注
→ 校验隔离
→ 去重与新旧判断
→ 披露锚对账
→ 冲突检测
→ 置信度初评
→ 重要性评分
→ 分流 / 入账 / 状态更新
```

这条链的产物不是“最终事实”，而是 L2 的观测证据资产。

## 3.5 轻处理与 B 类语义清洗

结构切分与 A 类规则确定性清洗已由 L1 完成。L2 的轻处理自已结构化的 `document_unit` 或轻载返回记录开始，统一适用于全部来源。

轻处理优先采用确定性方法，重点完成：

- 去重；
- 时间识别；
- 公司代码 / 名称匹配；
- 证券简称消歧；
- 资料类型分类；
- 是否公告 / 财报 / 纪要 / 研报 / 访谈 / 新闻 / 社媒；
- 语种识别。

B 类语义清洗 / 精简指需依据上下文判断信息价值、判定标准随时间演进、无法由稳定规则覆盖的处理。它是可选派生步骤：

- 输出为**版本化派生视图**，带 method / version，指回不可变 `document_unit`；
- 不修改 unit 本身；
- 下游 claim 记录其引用的派生视图版本，确保该 claim 当时所见文本可复现；
- 该步骤规模应保持最小，v1 可暂不实现、仅预留。

LLM 只在模糊文本、复杂切分、主体消歧、非结构化摘要、长文本信息抽取时介入。

## 3.6 claim 抽取

claim 是 L2 的基本单位。L2 要把资料拆成 claims，例如：

- 某公司某产品收入是多少；
- 某产品价格环比上涨；
- 某客户订单开始放量；
- 某产线良率改善；
- 某公司某项费用率下降；
- 某供应商与公司存在供应关系；
- 某技术路线可能替代另一技术路线；
- 某研究员预测明年净利润多少；
- 某公司管理层表示某产能将在某季度释放。

抽取时必须区分 claim 性质：

- 事实；
- 观点；
- 估算；
- 预测；
- 传闻；
- 关系；
- 解释；
- 假设。

不同性质的 claims 进入 L2 后，采信规则不同。比如，事实型披露锚可以直接成为高优先级观测证据；传闻型 claim 默认只能进入冷存或待核验；卖方预测可以作为观点型 / 预测型 claim，但不能伪装成事实。

## 3.7 L2 claim 的基本内容

一个 L2 claim 至少要表达清楚：

- 主体：公司、子公司、产品、客户、供应商、行业、区域、主题；
- 主体域：company / industry / macro / theme；
- 指标 / 事件：收入、毛利率、价格、订单、产能、良率、费用率、供应关系等；
- 陈述内容：数值或文本；
- 单位：元、美元、吨、片、台、bps、百分比等；
- 时间范围：某季度、某年度、某时点、未来期间；
- 口径标签：财报口径、扣非口径、产品口径、区域口径、含税 / 不含税、合并 / 单体等；
- 性质：事实、估算、观点、预测、传闻、关系、解释、假设；
- 来源链：来源类型、追溯等级、访问记录、原文定位；
- 来源层级：Tier 0A / 0B / 1 / 2 / 3 / F；
- 置信度；
- 冲突状态；
- 采信状态；
- 下游引用关系。

这不是数据库 schema，只是架构层必须保留的信息维度。

## 3.8 L2 的三类长期对象

### 3.8.1 原始观测 claims

原始观测 claims 是所有来源的并列陈述。它们不互相覆盖。

例如：

- A 研报认为某产品 2026 年毛利率 35%；
- B 纪要认为毛利率 30% 左右；
- 公司公告没有披露单品毛利率，但披露整体毛利率 28%；
- 专家访谈认为新产品初期良率较低。

这些都应该并列进入 L2，而不是只保留一个“最终答案”。

### 3.8.2 当前采信视图

当前采信视图是系统在某个时点认为最适合进入派生或预测的观测证据集合。

它不是第二套真相，而是对原始观测 claims 经过规则、对账、冲突检测、人工裁决后的当前视图。

当前采信视图必须版本化，并支持 as-of 查询。

### 3.8.3 人工裁决记录

人工裁决不应只体现在 claim 状态变更里。它本身也是系统长期资产。

每次裁决至少要说明：

- 裁决对象；
- 裁决结果；
- 理由；
- 使用资料；
- 是否覆盖机器判断；
- 影响哪些 claims；
- 是否触发 L3 失效；
- 是否触发 L5 重算；
- 是否应回写规则或框架。

## 3.9 claim 状态机

状态机必须存在，但枚举可以后续微调。当前建议：

| 状态 | 含义 |
|---|---|
| pending | 已入账，尚未采信或裁决 |
| accepted | 当前被采信，可进入 L3 派生或 L5 假设 |
| contested | 与其他证据、披露锚、派生结果或模型假设冲突 |
| rejected | 已被否定或不再适合采信 |
| archived | 历史保留，不再参与当前预测 |
| provisional | 临时采信，等待补证据或财报验证 |
| scenario-set | 不确定性过大，进入情景分支而非单点采信 |

高重要性 contested claim 不能被静默吞掉。它必须至少触发：

- 人工待办；
- L3 情景派生；
- L5 情景分支；
- 影子预测；
- 或正式预测阻塞。

## 3.10 关系型 claims

供应商、客户、产业链归属、竞争关系、份额、技术路线位置，如果是来源直接说的，应放在 L2。

例如：

- 某调研纪要说 A 公司进入某客户供应链；
- 某公告说 A 公司与 B 客户签订框架协议；
- 某专家说 C 产品目前主要由 D 厂商供货；
- 某研报认为某公司在某环节份额约 20%。

这些都是 source-backed claims，不属于 L4 静态知识。

原因很简单：这些关系会变，有来源，会冲突，会影响预测。

L4 只放结构模板，例如“光模块行业应该看客户结构、速率迭代、良率、产能、价格、DSP / 光芯片 / 封装 / 测试环节”。至于“某家公司当前对应哪个客户、哪个产品、哪个供应商”，应由 L2 关系型 claims 和 L3 派生视图动态生成。

## 3.11 披露锚对账

外部细分数据不能无解释地违反披露锚。

例如，卖方给出某公司分产品收入和毛利，如果分项加总与财报披露的业务收入差距过大，系统不能直接采信。它应当：

1. 识别可能口径差异；
2. 查找是否存在合并范围、期间、币种、税费、一次性项目、产品分类差异；
3. 按对账规则计算偏离程度；
4. 根据偏离程度调整置信度；
5. 超阈值转 contested；
6. 必要时生成待办或情景分支。

对账规则本身不放在 L2，它放在 L4 的规则工件库。L2 只是加载规则并执行观测层对账。

复杂对账后的结果，例如“可用于预测的分业务收入重构表”“对账后毛利率区间”，放在 L3。

## 3.12 校验隔离

生产 agent 和校验 agent 必须隔离。

- 生产 agent：负责从原文抽取 claim、给出初步结构化结果。
- 校验 agent：只看原文和抽取结果，不看生产 agent 的推理过程。
- 冲突 agent：对比历史 claims、当前采信视图、披露锚和相关假设。
- 重要性 agent：可以提出重要性建议，但最终是否执行由确定性准入规则决定。

目的不是迷信多 agent，而是降低单次抽取幻觉和顺着一个错误推理继续错下去的概率。

## 3.13 L2 分流结果

L2 的输出有五类：

1. **进入原始观测 claims**：有长期复用价值或可能影响预测；
2. **更新当前采信视图**：通过对账、冲突检测或人工裁决后成为当前可用证据；
3. **进入冷存 / 弱信号池**：单条不足以行动，但值得累计观察；
4. **进入人机待办**：需要人工裁决、补证据、问卖方、问专家、查公告；
5. **丢弃 / 垃圾池**：低质量、重复、无法定位主体、没有预测价值、明显噪声。

丢弃也要可粗略审计。至少要能知道某批资料为什么没有进入 L2。

## 3.14 L2 的 as-of 语义

L2 必须支持 as-of 查询：

> 回到某个历史时点，系统当时见过哪些观测 claims、采信哪些、否定哪些、忽略哪些、冲突哪些？

没有 as-of，就无法做真正复盘。因为不能用今天才知道的信息评价昨天的预测。

# 4. L3：派生证据层（Derived Evidence Layer）

## 4.1 L3 的定位

L3 是系统基于 L2 观测证据和 L4 规则 / 框架构造 derived claims 的层。

它负责回答：

> **系统基于来源证据、口径契约、对账规则、计算规则和框架映射，构造出了什么？**

L3 不直接抽取原文，不直接处理 PDF，不直接保存来源原文，也不直接发布预测。

它专门管理：

- derived claims；
- 归一化财务事实；
- 口径重构值；
- 披露锚对账后的指标；
- 分业务 / 分产品 / 分客户的重构视图；
- 行业汇总；
- 主题暴露度；
- 供应链 / 客户 / 技术路线派生图；
- 用于 L5 预测的中间事实层。

## 4.2 L3 不是第二套真相

L3 不允许变成一套脱离来源的“事实库”。

每条 derived claim 都必须能回答：

- 用了哪些 L2 claims；
- 用了哪个当前采信视图；
- 用了哪版口径契约；
- 用了哪些对账 / 计算 / 聚合规则；
- 用了哪些框架条目；
- 在哪个 as-of 时点生成；
- 如果输入变化，应该怎么失效和重算。

一句话：

```text
L2 = 来源说了什么；
L3 = 系统按规则从这些说法中构造出了什么。
```

## 4.3 L3 的输入

L3 的输入主要来自三类：

1. **L2 当前采信视图**：常规派生的主输入；
2. **L2 contested / scenario-set claims**：用于情景派生、区间派生、冲突解释；
3. **L4 规则与框架**：口径契约、对账规则、行业模板、驱动因子模板、聚合规则。

L3 原则上不直接吃 L1 原文。必要时可以通过 L2 claim 的来源链回查原文，但不能绕过 L2 直接把原文变成 derived claim。

## 4.4 L3 的主体域

L3 同样支持四类主体域：

| 主体域 | L3 派生例子 |
|---|---|
| company | 某公司 2026E 800G 收入区间；某公司扣非净利润重构；某产品毛利率基准值 |
| industry | 光模块 1.6T 渗透率路径；某行业价格指数重构；行业供需缺口 |
| macro | 汇率变动对某公司毛利率影响；出口管制对某产业链订单节奏影响 |
| theme | GB300 主题受益公司暴露度矩阵；CPO 技术路线公司映射；AI 服务器链条需求扩散图 |

这四类可以互相引用和传播，但不能混成无边界大图。L6 负责控制传播范围和重算成本。

## 4.5 L3 的典型 derived claims

### 4.5.1 口径重构型

由口径契约生成，例如：

- 从原始财报口径净利润派生扣非净利润；
- 把历史财务数据重构为新合并范围；
- 剔除一次性项目后的经营性利润；
- 按公司附录调整后的预测口径利润。

### 4.5.2 对账后事实型

由多个 L2 claims 与披露锚对账后生成，例如：

- 对账后的分业务收入表；
- 对账后的分产品毛利率区间；
- 分客户收入估算与财报总收入之间的桥；
- 卖方分项估算与公司披露总量之间的偏离解释。

### 4.5.3 计算派生型

由数值 claims 计算生成，例如：

- 毛利率；
- 费用率；
- ASP；
- 单位成本；
- 产能利用率；
- 收入同比 / 环比；
- 库存周转；
- 应收周转；
- 经营现金流转化率。

### 4.5.4 关系网络派生型

由 L2 关系型 claims 派生，例如：

- 某公司在某主题链条中的暴露度；
- 某公司对某客户 capex 的敏感度；
- 某产品供应链图；
- 某技术路线下的公司映射；
- 某行业上游瓶颈对下游公司的传导路径。

### 4.5.5 区间 / 情景派生型

由 contested claims 或不确定性输入派生，例如：

- 某产品毛利率合理区间；
- 某客户订单放量的乐观 / 基准 / 悲观路径；
- 某政策事件对收入确认节奏的情景矩阵；
- 某原材料价格对毛利率的敏感性表。

## 4.6 L3 derived claim 的基本内容

一个 L3 derived claim 至少要表达清楚：

- 派生主体：公司 / 行业 / 宏观 / 主题；
- 派生指标或事件；
- 派生值：数值、区间、文本、矩阵或结构化视图；
- 时间范围；
- 口径标签；
- 输入 claims 集合；
- 使用的 L2 当前采信视图版本；
- 使用的 L4 口径契约 / 对账规则 / 计算规则 / 框架条目；
- 派生方法和版本；
- as-of 时间；
- 置信度继承与调整逻辑；
- 是否含 contested 输入；
- 是否可进入 L5 主假设；
- 失效条件；
- 下游引用关系。

这同样不是数据库 schema，而是架构层必须保留的信息维度。

## 4.7 L3 的状态

L3 derived claims 至少要有状态概念：

| 状态 | 含义 |
|---|---|
| valid | 当前有效，可供 L5 使用 |
| stale | 输入 claim、采信视图、规则、契约或框架变化后已失效，等待重算 |
| contested | 派生结果内部或与披露锚 / 其他派生结果冲突 |
| provisional | 临时可用，等待补证据或财报验证 |
| scenario-derived | 由 scenario-set / contested 输入生成，只能用于情景分支或敏感性 |
| archived | 历史保留，不参与当前预测 |

L3 的状态变化应触发 L6 判断是否影响 L5 假设。

## 4.8 L3 与 L2 的边界例子

以供应链信息为例：

```text
L2：某调研纪要说 A 公司进入英伟达供应链。
L2：某公告说 A 公司与 B 客户签订框架协议。
L2：某研报说 A 公司在某环节份额约 20%。

L3：系统派生 A 公司在 GB300 光模块链条中的暴露度为高。
L3：系统派生 A 公司收入对北美 CSP capex 的敏感度较高。
L3：系统派生该公司 2026E 某产品收入区间。
```

以毛利率为例：

```text
L2：卖方 A 认为某产品毛利率 35%。
L2：纪要 B 认为某产品毛利率 30%。
L2：公司财报披露整体毛利率 28%。
L2：专家 C 认为新产品初期良率偏低。

L3：系统派生该产品当前可用毛利率区间 30%–35%。
L3：系统派生基准毛利率 32%。
L3：系统标记该 derived claim 含 contested 输入，需进入 L5 情景分支。
```

## 4.9 L3 到 L5 的使用方式

L5 原则上优先消费 L3，而不是直接消费 L2。

原因是 L3 已经完成：

- 口径统一；
- 披露锚对账；
- 来源冲突处理；
- 计算派生；
- 行业 / 主题映射；
- 版本化重构。

但 L5 不能完全看不到 L2。正式预测快照必须能从 L5 假设追到 L3 派生证据，再追到 L2 原始 claims 和 L1 source_ref / document_unit。

## 4.10 L3 的 as-of 语义

L3 必须支持 as-of 语义：

> 回到某个历史时点，系统当时基于当时可得的 L2 证据和当时的规则 / 契约，派生出了什么？

这对财报复盘非常关键。不能用后来的口径契约、后来的供应链信息、后来的客户结构去评价旧快照。

# 5. L4：知识与规则层

## 5.1 L4 的定位

L4 是旁路层，不是主管道。

它不装“世界事实”，而装：

1. **如何理解世界的框架**；
2. **如何操作资料、证据和预测的规则**。

L4 被 L2、L3、L5、L6 和课题行动加载使用。

## 5.2 框架库

框架库存放解释结构，例如：

- 行业研究框架；
- 产业链结构模板；
- 行业驱动因子模板树；
- 商业模式模板；
- 技术路线框架；
- 政策解读方法；
- 公司治理和管理层动机透镜；
- 财务预测模板；
- 三大表建模框架；
- 现金质量检查框架；
- 常见利润桥模板。

框架条目必须：

- 条目化；
- 版本化；
- 可引用；
- 可被快照记录；
- 可被复盘归因；
- 重大修改需要人批。

## 5.3 规则工件库

规则工件库存放操作规则，例如：

- 清洗规则；
- 抽取规则；
- 对账规则；
- 冲突仲裁原则；
- 置信度规则；
- 重要性准入规则；
- 派生规则；
- 聚合规则；
- 口径契约；
- 课题模板；
- 行动手册；
- 财报复盘归因规则；
- 正式快照发布规则。

规则和调度必须分离。

- 规则回答：应该怎么判断、怎么处理、怎么对账、怎么派生。
- 调度回答：什么时候执行、执行什么、由谁执行、执行后放到哪里。
- 行动回答：实际执行一次抽取、校验、重算、检索、派生、课题推进。

## 5.4 口径契约

口径契约归入规则工件库，但它是最高优先级规则之一。

它定义：

- 历史财务数据如何重构；
- 未来预测按什么口径输出；
- 财报季复盘按什么口径对比；
- 哪些调整进入净利润；
- 哪些调整只作为备注；
- 哪些项目需要公司附录；
- 哪些派生 claims 可以进入正式预测；
- 哪些派生 claims 只能作为补充说明。

口径契约采用：

```text
全局基准 + 公司附录
```

全局基准可以很轻：

- v0：原始财报口径；
- v0.1：原始财报口径 + 扣非；
- v0.2：增加常见一次性项目处理规则；
- 后续：公司特定附录。

每条契约政策最好拆成两段：

1. 文字理由：为什么这么处理；
2. 可执行规则：系统如何实际处理。

引擎只执行规则段，文字理由供人和 agent 理解。

硬规则：

> 历史重构、未来预测、财报复盘必须引用同一版口径契约。

如果契约升版，应触发相关 L3 derived claims、公司历史重算、模型重估和复盘重述，但不能覆盖原预测快照。

## 5.5 L4 不装事实

玩家、客户、供应商、竞争关系、份额、技术路线位置，不应直接放在 L4 作为静态事实。

它们应当这样放：

```text
来源直接说的关系 → L2 关系型 claims；
系统基于多条关系 claims 形成的供应链图 / 暴露度 / 技术路线映射 → L3 派生证据；
“应该从哪些维度理解这个行业” → L4 框架条目。
```

这条边界必须守住。否则 L4 会变成一个混杂事实、观点、规则和框架的知识库，后续无法复盘、无法失效、无法重算。

## 5.6 可训练大脑

系统不是静态规则堆，而是一个能通过复盘不断变好的“投资大脑”。这个想法在架构上落在 L4。

但它不是一开始就训练大模型，而是先建立可学习结构：

```text
框架条目 / 规则工件
→ 被 L2 / L3 / L5 引用
→ 产生预测快照
→ 财报验证
→ 误差归因到条目
→ 形成 diff 提案
→ 历史重放评测
→ 人批晋级
```

类比深度学习：

- 前向传播：框架 + 规则 + 观测证据 + 派生证据 → 假设 → 预测；
- loss：预测与财报实际之间的误差；
- 反向归因：定位是哪个来源、规则、框架条目、派生逻辑、假设或人工覆盖导致误差；
- 参数更新：修改框架条目、规则工件、契约附录、预测模板；
- 冻结层：系统边界、原始观测 claims、执行日志、快照不可回改、调度原则；
- 可训练层：行业模板、规则工件、公司附录、托底规则、课题模板。

早期训练不是自动化模型训练，而是“版本化文本 + 人审 + 历史重放”。数据积累后，再考虑自动评测、模板推荐、甚至微调。

## 5.7 共享与公司特例

每家公司每年只有 4 个财报验证点。逐家公司学习样本太少。

因此，行业共性必须尽量进入共享模板条目，公司特例才进入公司附录。

例如：

- “光模块公司收入预测需要拆客户、速率、产品代际、价格、产能、良率”应进入行业共享模板；
- “某公司某客户供货节奏特殊”应进入公司模型或公司附录；
- “某行业玩家名单”不进框架层，应由 L2 关系型 claims 和 L3 派生视图生成。

# 6. L5：公司预测层

## 6.1 L5 的定位

公司预测层负责把观测证据、派生证据、规则、框架和人工判断转化为公司未来财务预测。

当前阶段聚焦净利润，但内部不能只存一个净利润数字。必须保留利润桥、关键假设、现金质量检查和中间节点。

## 6.2 L5 的输入优先级

L5 原则上优先消费 L3 派生证据。

典型输入顺序是：

```text
L2 原始观测 claims
→ L2 当前采信视图
→ L3 派生证据 / 对账后事实 / 归一化事实
→ L5 假设账本 / 驱动树 / 利润桥
```

但 L5 的正式快照必须能追溯到 L2 和 L1：

```text
forecast_node
→ assumption
→ derived claim
→ observed claim
→ source_ref / document_unit
```

没有这条链，预测无法解释，也无法复盘。

## 6.3 每家公司一个预测对象

每家公司对应一个预测对象，内部至少包括：

1. 驱动树；
2. 利润桥；
3. 假设账本；
4. 数据绑定与订阅；
5. 现金质量检查；
6. 情景分支；
7. 工作底稿；
8. 正式预测快照；
9. 影子预测快照；
10. 财报季复盘记录。

## 6.4 驱动树

驱动树是公司财务预测的结构。

净利润阶段也不能只预测净利润。至少要有完整利润桥：

```text
收入
→ 成本
→ 毛利
→ 费用
→ 经营利润
→ 利润总额
→ 所得税
→ 少数股东损益
→ 归母净利润
```

每个节点都要知道：

- 这个节点来自哪个框架条目；
- 用了哪些 L3 derived claims；
- 能追到哪些 L2 observed claims；
- 用了哪些假设；
- 是否是托底规则；
- 是否有冲突；
- 是否有人工覆盖；
- 下次验证点是什么。

## 6.5 托底规则

早期不可能每个科目都研究到位。因此允许托底规则，例如：

- 历史均值；
- 最近趋势；
- 行业均值；
- 管理层指引；
- 卖方一致估算；
- 人工临时假设。

但托底规则必须显式标记，不能伪装成深度研究。

系统要能回答：

> 这次净利润预测里，哪些节点是研究化 driver，哪些节点只是托底？

长期演进方向是逐步把高影响托底节点替换为研究化 driver。

## 6.6 假设账本

假设账本不是 L2，也不是 L3。

- L2 记录来源直接说了什么；
- L3 记录系统派生出了什么；
- 假设账本记录模型当前采用什么输入。

一个假设至少要说明：

- 对应公司；
- 对应业务 / 产品 / 客户 / 区域；
- 对应指标；
- 对应期间；
- 假设值；
- 使用的 L3 derived claims；
- 能追溯到的 L2 observed claims；
- 使用的口径契约；
- 使用的框架条目；
- 是否人工覆盖；
- 置信度；
- 冲突状态；
- 下次验证点；
- 对净利润敏感度。

## 6.7 数据绑定与订阅

预测节点必须注册自己关心什么数据。

绑定键可以概念化为：

```text
主体 × 业务/产品/客户/区域 × 指标/事件 × 期间 × 口径 × 来源优先级
```

订阅可以指向两层：

- 优先订阅 L3 derived claims，例如“对账后某产品毛利率”；
- 必要时订阅 L2 observed claims，例如“任何关于某客户订单放量的高 Tier 新观测 claim”。

当 L2 或 L3 出现匹配变化，L6 才能知道它可能影响哪个假设、哪个节点、哪个公司快照。

## 6.8 现金质量检查

即使当前正式输出只是净利润，也必须保留现金质量检查。

原因是净利润容易被收入确认、应收、存货、合同负债、资本化、减值、一次性项目污染。

现金质量检查至少包括：

- 应收账款；
- 存货；
- 合同负债；
- 经营现金流；
- 资本开支；
- 折旧摊销；
- 资产减值；
- 费用资本化；
- 非经常性损益。

它不一定一开始影响净利润点值，但必须影响置信度、污染标记、待办优先级和复盘归因。

## 6.9 情景分支

情景分支只用于高重要性 contested driver。

例如：

- 某产品毛利率有两个可信来源但差异很大；
- 某大客户订单是否放量不确定；
- 某产品价格是报价还是成交价不清楚；
- 某产能是否按期释放存在分歧。

情景分支不能无限扩散。必须限制：

- 分支数量；
- 分支变量数量；
- 分支持续时间；
- 是否影响正式预测；
- 是否需要人工裁决。

主预测使用当前采信假设；情景分支用于预测区间、敏感性和待办排序。

## 6.10 正式快照与影子快照

- **影子快照**：系统可以自动生成，用于观察、比较、提前发现变化。
- **正式快照**：进入对外正式输出，需要通过发布闸门。

正式快照必须保存：

- L2 观测证据视图版本；
- L3 派生证据版本；
- 口径契约版本；
- 框架条目引用；
- 规则工件版本；
- 模型版本；
- 假设版本；
- 中间计算节点；
- 利润桥；
- 现金质量检查；
- 情景分支；
- 敏感变量；
- 最脆弱假设；
- 人工覆盖记录；
- 发布理由；
- 未解决冲突。

发布后不可回改。

# 7. L6：调度脊柱

## 7.1 定义

调度脊柱不是研究方法本身。它负责让系统持续运行。

结构是：

```text
感知 → 触发 → 准入 → 行动 → 队列
```

## 7.2 感知：最小血缘图 + 订阅索引

第一阶段不要把血缘图做得过重。目标不是完整知识图谱，而是回答六个工程问题：

1. 某个 source_ref / document_unit 变化，会影响哪些 L2 claims？
2. 某条 L2 claim 或当前采信视图变化，会影响哪些 L3 derived claims？
3. 某个口径契约升版，会影响哪些 L3 derived claims 和预测公式？
4. 某个框架条目变化，会影响哪些 L3 派生逻辑、公司模型和假设？
5. 某个 L3 derived claim 变化，会影响哪些 L5 假设？
6. 某个假设变化，会影响哪些预测节点和快照？

因此最小血缘边先锁定：

```text
source_ref / document_unit → observed_claim
observed_claim → current_evidence_view
current_evidence_view / observed_claim → derived_claim
contract / rule_artifact / framework_entry → derived_claim
contract / framework_entry / rule_artifact → driver_tree / assumption / action
research_topic → produced observed_claims / derived_claims / assumptions / framework_diffs / rule_diffs
observed_claim / derived_claim → assumption
assumption → forecast_node
forecast_node → snapshot
```

其他关系先通过执行日志保留，不急着建复杂图。

## 7.3 触发

触发来源包括三类。

### 7.3.1 事件触发

- 新公告；
- 新财报；
- 新研报；
- 新纪要；
- 新 API 数据；
- 新 web 检索结果；
- 新 L2 observed claim；
- L2 claim 状态变化；
- L2 当前采信视图变化；
- 新 L3 derived claim；
- L3 derived claim 失效或重算；
- 人工裁决；
- 口径契约升版；
- 框架条目变更；
- 规则工件变更；
- 财报披露。

### 7.3.2 定时触发

- 冷存池扫描；
- 弱信号累计扫描；
- 假设过期检查；
- 派生证据过期检查；
- 模型陈旧检查；
- 临近财报检查；
- 失败任务重试；
- 影子快照定期重算。

### 7.3.3 人工触发

人工触发是一等触发源，例如：

- 新增个股研究；
- 开启某个产品毛利率课题；
- 要求重算某公司预测；
- 强制查看某条冲突；
- 手动裁决一个假设；
- 让系统生成调研待办。

人工触发可以直通，但仍要经过准入记录：系统需要知道这件事为什么被执行。

## 7.4 准入

准入不是一个复杂“智能体”，而是触发和行动之间的策略层。

作用是：

1. 控成本；
2. 控噪声；
3. 控人的注意力；
4. 保证事后可审计。

候选行动根据重要性映射到：

- 立即执行；
- 低优先入队；
- 仅记录；
- 转人工确认；
- 丢弃。

重要性特征包括：

- 来源 Tier；
- 追溯等级；
- 影响公司数量；
- 涉及预测期间；
- 与当前假设偏离程度；
- 对净利润敏感度；
- 是否临近财报；
- 是否涉及高置信冲突；
- 是否来自硬披露锚；
- 是否解决已有待办；
- 是否导致 L3 派生证据失效；
- 是否会改变正式快照候选。

LLM 可以对定性材料打辅助分，但不能单独决定准入结果。

## 7.5 行动库

行动分为原子行动和复合行动。

### 7.5.1 原子行动

- 获取资料；
- 登记来源；
- 归档公告 PDF；
- 文本抽取；
- 表格抽取；
- claim 抽取；
- 校验；
- 对账；
- 冲突检测；
- 更新 L2 claim 状态；
- 更新当前采信视图；
- 生成 / 重算 L3 derived claim；
- 标记 derived claim stale；
- 更新假设；
- 重算预测节点；
- 生成影子快照；
- 生成正式快照候选；
- 生成待办；
- 冷存归档；
- 检索未得记录。

### 7.5.2 复合行动

- 入库流程；
- 观测证据变更传播；
- 派生证据重算；
- 研究课题；
- 个股初始化；
- 财报季复盘；
- 口径契约升版；
- 框架条目重放评测；
- 来源可信度更新。

每个行动必须声明：

- 输入；
- 输出；
- 加载的规则工件；
- 使用的框架条目；
- 执行者：代码 / agent / 人；
- 日志记录内容；
- 失败后怎么处理。

## 7.6 失效队列

当上游变化后，系统不全量重算，而是：

```text
定位受影响对象
→ 判断重要性
→ 标记失效
→ 入失效队列
→ 自动重算 / 影子重算 / 人工确认 / 仅观察
→ 生成新快照或待办
```

典型失效链路：

```text
L1 新资料 / 新 source_ref
→ L2 新增或更新 observed claims
→ L2 当前采信视图变化
→ L3 相关 derived claims stale
→ L3 重算派生证据
→ L5 相关假设 stale
→ L5 生成影子预测
→ 必要时进入人工待办或正式快照候选
```

如果是规则变化：

```text
L4 口径契约 / 对账规则 / 行业框架变化
→ L3 derived claims stale
→ L5 假设和预测节点重检
→ 影子快照 / 复盘重述
```

如果是人工裁决变化：

```text
人工裁决某个 contested claim
→ L2 当前采信视图变化
→ L3 派生结果重算
→ L5 相关预测节点重算
```

失效不是错误，而是系统正常运行状态。

# 8. 研究课题：拉式回路

## 8.1 为什么必须有课题

成熟公司模型可以依赖数据流入后持续更新。

但新增个股、冷启动、深度 driver 建设、冲突裁决，不能只靠被动信息流。很多研究是从预测问题出发，一层层追下去：

```text
净利润预测不稳
→ 毛利率假设不稳
→ 某产品毛利不清楚
→ 成本结构不清楚
→ 原材料价格、良率、工艺、客户结构不清楚
→ 需要主动找资料 / 问专家 / 查公告 / 查竞品
```

这就是课题。

## 8.2 课题定义

课题是复合行动，必须绑定目标变量。

一个课题至少包含：

- 目标公司 / 行业 / 主题；
- 目标变量；
- 阻塞问题；
- 预测影响；
- 当前 L2 证据缺口；
- 当前 L3 派生缺口；
- 信息需求清单；
- 可用来源路径；
- 子课题；
- 预算限制；
- 预期产出；
- 结案标准。

禁止无边界课题，例如“研究一下 AI 光模块”。

允许的写法是：

> 为了更新中际旭创 2026E 毛利率假设，研究 800G / 1.6T 产品价格、良率、客户结构和产能释放对产品毛利的影响。

## 8.3 课题来源

课题可以来自：

- 人工主动提出；
- 新增个股初始化；
- 模型发现关键假设缺失；
- 高重要性 L2 claim 冲突；
- L3 派生结果含 contested 输入；
- 弱信号池累计；
- 财报复盘发现误差来源；
- 框架条目缺失；
- 数据流自动提出候选，等待人批准。

## 8.4 课题执行循环

立题第一步是判定课题类型，因为类型决定执行顺序：

- **值课题**：目标节点的驱动树结构已经建好，本次只刷新取值或采信，不动结构。走**证据先行**。
- **结构课题**：本次要新建、展开（托底标量 → 子树）或修改驱动树结构 / 拆解口径。走**结构先行**——结构决定了要取哪些数。

值课题（证据先行）：

```text
立题（结构已在，只刷新取值 / 采信）
→ 生成信息需求清单
→ 查 L2 观测证据 → 查 L3 派生证据 → 查冷存池
→ 主动获取资料 → L1 登记 → L2 形成 claims
→ 对账和冲突检测
→ L3 更新派生证据
→ 更新假设（只更新节点取值，不改结构）
→ 重算 → 生成影子预测
→ 必要时开子课题
→ 结案或转人工待办
```

结构课题（结构先行）：

```text
立题（要新建 / 展开 / 改驱动树结构）
→ 生成信息需求清单（含“该怎么拆”的结构问题）
→ 查 / 定 L4 框架模板与口径契约（已有则引用；缺失则确定，新的共享结构走人批）
→ 据模板改 L5 驱动树骨架，新节点登记数据绑定
→ 查 L2 观测证据 → 查 L3 派生证据 → 查冷存池
→ 主动获取资料 → L1 登记 → L2 形成 claims
→ 对账和冲突检测
→ L3 形成 / 更新派生证据
→ 节点取值，更新假设 / 驱动树（必要时提框架 / 规则 diff）
→ 重算 → 生成影子预测
→ 必要时开子课题
→ 结案或转人工待办
```

关键硬规则：

1. 课题主动获取的资料，必须和被动资料一样进入 L1、L2、L3，不能直接绕过证据层改预测。
2. 立题必须先判定值课题 / 结构课题。
3. 结构课题必须结构先行：先查 / 定 L4 模板与口径、再改 L5 驱动树骨架，然后才取数。
4. 结构课题若需新建或修改共享框架条目，结构步必须经人批；值课题只刷新取值，不触发框架人批。

## 8.5 课题预算

课题必须控制递归和成本。

需要限制：

- 最大子课题深度；
- 最大子课题数量；
- 最大检索轮次；
- 最大 agent 执行次数；
- 最大人工待办数量；
- 是否允许影响正式预测；
- 结案时间或复查时间。

否则系统会无限研究，无法产出预测。

## 8.6 新增个股初始化

新增个股是最大的标准课题。

初始化目标不是“一次性研究完”，而是形成：

> **能出影子预测的最小公司模型。**

初始化课题至少覆盖：

1. 公司主营业务；
2. 产品 / 服务拆分；
3. 收入结构：产品、客户、区域；
4. 成本结构；
5. 毛利率拆分；
6. 费用结构；
7. 上下游关系；
8. 供应商与客户；
9. 产能与扩产；
10. 竞争格局；
11. 技术路线；
12. 价格驱动；
13. 历史财务重构；
14. 现金质量；
15. 管理层诉求和激励；
16. 主要争议点；
17. 最小预测驱动树；
18. 未解决待办。

未研究透的部分允许托底，但要显式标记。

# 9. 标准流程

## 9.1 入库流程

```text
资料进入 / 访问记录生成
→ L1 登记来源与追溯等级
→ L1 文档结构切分 + A 类规则确定性清洗
→ L2 接收 document_unit / source_ref
→ L2 轻处理
→ L2 B 类语义清洗（可选，生成版本化派生视图）
→ L2 claim 抽取
→ L2 校验隔离
→ L2 对账和冲突检测
→ L2 重要性评分
→ L2 分流到原始观测 claims / 当前采信视图 / 冷存 / 待办 / 丢弃
→ L6 判断是否触发 L3 派生或 L5 行动
```

## 9.2 观测证据变更传播流程

```text
L2 claim 新增 / 状态变化 / 人工裁决 / 当前采信视图变化
→ L6 查订阅与最小血缘
→ 定位受影响 L3 derived claims
→ 准入判断
→ 标记 L3 对象 stale 或进入重算队列
→ L3 重算 / 情景派生 / 生成待办
→ 判断是否影响 L5 假设
```

## 9.3 派生证据重算流程

```text
L3 derived claim 失效或新建需求
→ 加载 L2 当前采信视图和必要 contested claims
→ 加载 L4 口径契约 / 对账规则 / 框架条目
→ 执行派生 / 对账 / 聚合 / 计算
→ 生成新的 derived claim 版本
→ 更新下游引用关系
→ L6 判断是否触发 L5 假设和预测节点重算
```

## 9.4 主动课题流程

```text
预测问题或人工需求
→ 立题
→ 查 L2 观测证据、L3 派生证据和冷存
→ 主动获取资料
→ 新资料标准入库
→ 更新 L2 claims 和 L3 derived claims
→ 更新假设和驱动树
→ 重算预测
→ 结案记录进入 M-LOG
```

## 9.5 新增个股流程

```text
人工指定公司
→ 启动个股初始化课题
→ 建立最小公司档案
→ 建立初始驱动树
→ 接入公告 PDF 和历史财报
→ 形成 L2 观测证据基础
→ 形成 L3 历史财务重构和利润桥
→ 形成当前核心假设
→ 标记托底节点和缺口
→ 生成影子预测
→ 形成待办列表
```

## 9.6 财报季复盘流程

```text
财报 / 公告入库
→ 作为硬披露锚进入 L2
→ L3 按快照当时口径契约重构实际值
→ 对比正式快照和影子快照
→ 按利润桥逐节点定位偏离
→ 做误差归因
→ 回写来源可信度、规则、框架、派生逻辑、假设、待办
→ 必要时提出框架 / 契约 / 规则 diff
```

复盘不能只看净利润误差。必须定位：

- 收入错了；
- 毛利率错了；
- 费用率错了；
- 税率错了；
- 少数股东损益错了；
- 一次性项目错了；
- 口径错了；
- 披露理解错了；
- 数据源错了；
- 抽取错了；
- L2 仲裁错了；
- L3 派生逻辑错了；
- 框架结构错了；
- 人工覆盖错了；
- 蒙对了但逻辑错了。

## 9.7 口径契约升版流程

```text
发现口径问题
→ 提出契约 diff
→ 人工审核
→ 小范围历史重算
→ 检查影响 L3 derived claims、公司模型和预测
→ 契约升版
→ 受影响 derived claims 和预测公式失效
→ 生成新快照或重述复盘
```

契约升版不能悄悄改变历史快照。

## 9.8 框架条目演进流程

```text
财报误差 / 课题发现 / 人工判断
→ 定位框架条目问题
→ 提出框架 diff
→ as-of 历史重放评测
→ 人工批准
→ 框架条目升版
→ 受影响 L3 派生逻辑、公司模型和假设重检
```

# 10. 冷存 / 弱信号池

## 10.1 定位

冷存池收纳暂时不足以进入当前采信视图或不足以影响预测的信息。

典型信息：

- 单条社媒传闻；
- 来源弱但方向有价值的线索；
- 暂时无法确认主体的信息；
- 与预测相关但重要性不足的资料；
- 暂时没有模型节点接收的信息；
- 查询未得记录；
- 低置信但可能成为课题种子的观察。

冷存池逻辑上属于 L2 观测证据体系的边缘区。它不是 L3，也不是垃圾箱。

## 10.2 触发机制

冷存必须有触发机制：

- 同主体同方向累计超过阈值；
- 与当前高敏感假设相关；
- 新硬锚出现后可验证；
- 被课题查询命中；
- 某个 driver 转 contested；
- 临近财报；
- 人工手动激活。

触发后，冷存信息可以：

- 重入 L2；
- 成为 L2 observed claim；
- 触发 L3 派生；
- 生成课题候选；
- 生成待办；
- 继续冷存。

## 10.3 Tier 3 的处理

自媒体、大 V、论坛、社群传闻原则上先进冷存。

它们可以作为：

- 早期信号；
- 检索线索；
- 课题触发器；
- 与其他证据的辅助印证。

但不能单独支撑正式预测里的 accepted claim 或 valid derived claim。

# 11. 人机待办

## 11.1 定位

人机待办不是普通 todo list，而是系统把“不确定性”和“研究缺口”转化为人类行动的界面。

待办必须落到变量级，不能只写“研究一下某公司”。

## 11.2 待办排序

排序公式可以概念化为：

```text
预测影响 × 置信度缺口 × 财报临近度 × 可解决性 × 分歧价值
```

其中：

- 预测影响：解决后可能改变多少净利润预测；
- 置信度缺口：当前证据距离正式采信差多少；
- 财报临近度：越接近验证点越优先；
- 可解决性：是否能通过公告、API、专家、卖方快速解决；
- 分歧价值：不同情景下净利润差异多大。

## 11.3 人类负责什么

人类主要负责：

- 批准新增个股初始化；
- 批准高成本课题；
- 裁决高重要性冲突；
- 批准口径契约变化；
- 批准核心框架变化；
- 做物理世界调研；
- 问卖方、专家、公司；
- 判断是否发布正式快照；
- 覆盖机器假设并给出理由。

## 11.4 机器负责什么

机器主要负责：

- 整理资料；
- 抽取 L2 claims；
- 做初步校验；
- 发现冲突；
- 对账；
- 生成 L3 derived claims；
- 标记派生证据失效；
- 建议待办；
- 生成课题信息需求清单；
- 查 L2 / L3 / 冷存；
- 主动检索；
- 生成影子预测；
- 生成复盘归因草稿；
- 提出框架 / 规则 diff。

## 11.5 人工裁决记录

每次人工裁决都要记录：

- 裁决对象；
- 裁决结果；
- 理由；
- 使用资料；
- 是否覆盖机器判断；
- 影响哪些 L2 claims；
- 影响哪些 L3 derived claims；
- 影响哪些假设；
- 是否触发重算；
- 是否应回写规则或框架。

人工判断不是系统外事件。它本身也是系统学习资料。

# 12. 执行日志、版本和快照

## 12.1 执行日志 M-LOG

执行日志记录系统做过什么。

至少包括：

- 何时执行；
- 由什么触发；
- 执行了什么行动；
- 输入是什么；
- 输出是什么；
- 加载了哪些规则工件；
- 引用了哪些框架条目；
- 使用了哪些 source_ref；
- 产生了哪些 observed claims；
- 产生或重算了哪些 derived claims；
- 改了哪些假设；
- 是否进入失效队列；
- 是否生成快照；
- 是否生成待办；
- 是否失败。

执行日志是后续补血缘、做复盘、训练框架的基础。

## 12.2 版本引用

正式快照和关键中间结果至少要记录：

- L2 观测证据视图版本；
- L3 派生证据版本；
- 口径契约版本；
- 框架条目版本；
- 规则工件版本；
- 模型版本；
- 假设版本；
- 数据源追溯等级；
- 执行日志 ID；
- 发布人 / 发布动作。

## 12.3 不可变与可变

不可变：

- 自维护 raw 原文；
- source_event / source_ref 历史记录；
- L2 原始观测 claims；
- L3 已生成的 derived claim 历史版本；
- 执行日志；
- 发布后的预测快照；
- 财报复盘原始记录。

可变视图：

- L2 当前采信视图；
- L3 当前有效派生证据视图；
- 当前假设；
- 当前预测模型实例；
- 当前待办状态；
- 当前框架最新版；
- 当前规则最新版。

“可变视图”变化时必须能追到历史版本。

## 12.4 快照不可回改

如果发现旧预测错了，不改旧快照。

正确做法是：

- 生成新快照；
- 或生成重述复盘；
- 或标记旧快照受某个后验信息影响；
- 但不覆盖原始发布记录。

# 13. 冲突、置信度和情景管理

## 13.1 冲突类型

系统至少识别：

- 数值冲突；
- 时间冲突；
- 口径冲突；
- 主体冲突；
- 来源冲突；
- 事实与观点混用；
- 估算和披露锚冲突；
- L2 观测 claim 与 L3 派生结果冲突；
- 预测与实际披露冲突；
- 框架解释冲突；
- 人工判断与机器判断冲突。

## 13.2 置信度来源

置信度不能只由 LLM 打分。

它应该综合：

- 来源 Tier；
- 追溯等级；
- 与硬披露锚的一致性；
- 是否多源印证；
- 来源历史准确率；
- 信息是否直接；
- 是否转述；
- 是否有明确期间和口径；
- 抽取校验是否通过；
- L3 派生是否依赖 contested 输入；
- 是否被财报验证；
- 是否有人裁决。

## 13.3 冲突处理

冲突处理不是简单选一个。

处理方式包括：

- 降置信；
- 转 contested；
- 生成待办；
- 生成情景分支；
- 暂时采信硬锚；
- 等待新资料；
- 请求人工裁决；
- 阻塞正式快照；
- 只允许影子预测。

## 13.4 情景分支

情景分支只服务于预测区间和不确定性表达，不是让系统逃避判断。

系统仍然需要有当前主假设。情景分支用于表达：

- 关键变量存在分歧；
- 不同分歧对净利润影响很大；
- 暂时没有足够证据裁决；
- 需要把不确定性显性化。

# 14. 财报季复盘与系统学习

## 14.1 财报是系统的验证集

财报季复盘不是写点评，而是系统训练和纠错过程。

财报披露后，系统要回答：

- 预测错在哪里；
- 是哪个节点先偏离；
- 是 L1 来源问题、L2 抽取 / 采信问题、L3 派生问题、L4 规则 / 框架问题、L5 假设问题，还是人工判断问题；
- 哪些规则应该更新；
- 哪些来源应该降权；
- 哪些公司模型应该修改；
- 哪些待办应该长期保留。

## 14.2 逐节点复盘

不能只看净利润误差。

必须沿利润桥复盘：

```text
收入
→ 毛利率
→ 毛利
→ 费用
→ 经营利润
→ 财务费用 / 投资收益 / 其他收益
→ 税率
→ 少数股东损益
→ 归母净利润
```

现金质量也要同步看：

- 利润是否转化为现金；
- 应收是否异常；
- 存货是否异常；
- 合同负债是否验证需求；
- 资本开支是否验证扩产；
- 是否出现减值或一次性项目。

## 14.3 误差归因分类

建议初始分类：

1. 来源错误；
2. 抽取错误；
3. 口径错误；
4. 对账失败；
5. L2 仲裁错误；
6. L3 派生逻辑错误；
7. 假设方向错误；
8. 假设节奏错误；
9. 模型结构错误；
10. 框架条目错误；
11. 规则工件错误；
12. 人工覆盖错误；
13. 突发事件；
14. 公司披露变化；
15. 蒙对但逻辑错误；
16. 蒙错但方向合理。

## 14.4 回写

复盘结果要回写到：

- 来源可信度；
- 抽取规则；
- 对账规则；
- 派生规则；
- 口径契约；
- 框架条目；
- 公司驱动树；
- 假设账本；
- 课题模板；
- 人机待办排序；
- 未来快照发布规则。

复盘如果不回写，系统不会变好。

# 15. 当前阶段的工程取舍

这不是路线图，而是当前架构必须承认的现实边界。

## 15.1 L1 先轻后重

当前不需要先做全量 raw data lake。

优先做：

1. 公告 / 财报 PDF 自维护；
2. 资料来源登记；
3. 查询和访问记录；
4. 追溯等级；
5. 与 L2 / L3 的证据链。

其他资料通过 API、MCP、web、人工上传进入，不强求原文级保存。

## 15.2 先把 L2 / L3 / L5 做硬

比起大而全的数据层，早期更重要的是：

- L2 observed claim 能入账；
- L2 claim 能追到来源记录；
- L2 claim 能冲突、能采信、能被人工裁决；
- L3 derived claim 能说明输入、规则、口径和 as-of；
- L3 派生结果能失效、重算、被 L5 引用；
- 假设能影响预测；
- 预测能生成快照；
- 快照能复盘。

## 15.3 baseline claims 暂不启用

一致预期、卖方预测、市场 consensus 可以以后作为对照模块。

当前阶段不把它放入核心流程，避免系统被一致预期牵着走。

如果后续启用，应作为 L2 的预测型观测 claims 和 L3 的 baseline 派生视图进入，不应直接进入 L5 主假设。

## 15.4 不依赖实时在线

系统按事件语义设计，但当前运行形态可以是：

- 定时批处理；
- 人工拉起；
- session 式 agent harness；
- 局部重算；
- 影子快照常算；
- 正式快照人工过闸。

不要求第一天实时流式运行。

## 15.5 harness 无关

可以用 Claude、Codex、MCP、Python、数据库、搜索工具等执行行动，但系统控制逻辑不能绑定某个 harness。

控制逻辑应该沉淀在：

- 框架条目；
- 规则工件；
- 口径契约；
- 行动日志；
- 快照约定；
- 待办机制。

# 16. 硬边界

以下边界建议锁定。

1. 系统唯一正式产出是公司未来财务预测；当前阶段是契约口径下净利润预测。
2. 系统不做估值、选股、买卖建议、便宜判断、组合构建。
3. L1 是资料资产层，不是必须自有全量 raw data 仓库。
4. 上市公司公告 / 财报 PDF 是自维护硬锚，原文尽量保存。
5. API、MCP、web、购买数据服务可以只保留来源标识、查询参数、访问时间和返回摘要。
6. 所有进入预测的证据必须有追溯等级；弱追溯必须有置信度上限。
7. L2 是观测证据层，负责 source-backed claims、当前采信视图、冲突状态、关系型 claims 和人工裁决。
8. L1 → L2 的入库处理只是 L2 的入口动作，不再单独作为长期层名称。
9. L3 是派生证据层，专门管理 derived claims、归一化事实、对账后事实和跨主体域派生视图。
10. derived claims 必须记录输入 claims、当前采信视图、规则 / 契约 / 框架版本和 as-of 时间。
11. 公司、行业、宏观、主题是 L2 / L3 的主体域，不是四套独立架构。
12. 事实、观点、估算、预测、传闻、关系、口径、框架、规则、假设、输出必须分开。
13. 披露为锚，分硬披露锚和软披露锚；外部数据必须与锚解释得通。
14. 对账规则放在规则工件库，L2 / L3 / L6 只是加载执行。
15. 口径契约第一天存在，可以很轻；采用全局基准 + 公司附录。
16. 历史重构、未来预测、财报复盘必须引用同一版口径契约。
17. 知识与规则层是旁路；框架库和规则工件库分开，但都条目化、版本化、可引用。
18. 玩家、客户、供应商、竞争关系：来源直接支持的放 L2；系统派生出的供应链图 / 暴露度 / 技术路线映射放 L3；理解框架放 L4。
19. L5 优先消费 L3，但必须能回溯 L2 和 L1。
20. 预测快照必须记录 L2 观测证据视图、L3 派生证据版本、框架条目和规则工件引用。
21. 账本、日志和快照必须支持 as-of 复盘。
22. 净利润阶段也必须保留完整利润桥，不允许只有一个净利润数字。
23. 现金质量检查必须存在，即使早期只影响置信度和待办。
24. 冲突不能吞掉；高重要性冲突阻塞正式快照，但不阻塞影子快照。
25. 情景分支只给高重要性 contested driver，且数量受限。
26. 能确定性完成的任务不交给 LLM；LLM 评分只能作为准入特征之一。
27. 生产和校验隔离；校验 agent 只看原文和产出，不看生产推理。
28. 调度脊柱采用感知 / 触发 / 准入 / 行动 / 队列结构。
29. 血缘第一阶段按最小集建设，其余依赖由执行日志保留。
30. 上游变化通过订阅和血缘局部重算，不做无脑全量重算。
31. 课题是复合行动，不是新层；必须绑定目标变量。
32. 课题主动获取的资料必须进入 L1 / L2 / L3，不能绕过证据层直接改预测。
33. 冷存 / 弱信号池必须有触发机制，否则只是垃圾箱。
34. 人工裁决必须记录理由和影响路径，并回流为规则和框架优化输入。
35. 财报季复盘是流程，不是报告；必须逐节点归因并回写上游。
36. 一致预期 baseline 暂时作为可选槽位，不进入第一阶段核心。
37. demo 期允许定时 / 人工拉起的批式运行，不要求实时在线。
38. 系统控制逻辑必须 harness 无关。
39. 快照发布后不可回改，只能生成新快照或重述复盘。

# 17. 术语表

| 术语 | 含义 |
|---|---|
| 资料资产层 | 负责资料访问、来源登记、披露锚归档、检索记录的层 |
| 披露锚 | 财报、公告、问询函等高优先级披露资料 |
| source_ref | 对某次来源访问、文件、API 查询或外部结果的可追溯引用 |
| document_unit | 自维护文档被切分后的最小可追溯单元 |
| observed claim | 来源直接支持的一条可判断陈述，属于 L2 |
| 当前采信视图 | 系统在某个时点认为可用于派生或预测的 L2 claims 视图 |
| 关系型 claim | 关于客户、供应商、竞争关系、技术路线、产业链位置等关系的 L2 claim |
| derived claim | 由 L2 claims 通过口径契约、对账规则、计算规则或框架映射派生的 L3 claim |
| 派生证据层 | 管理 derived claims、归一化事实、对账后事实和跨主体域派生视图的层 |
| 主体域 | company / industry / macro / theme，用于组织 L2 / L3 claims 的作用范围 |
| 追溯等级 | 某条证据能追溯到原文、数据服务、网页、人工记录或弱来源的程度 |
| 口径契约 | 规定历史重构、未来预测、财报复盘按什么口径处理的规则工件 |
| 框架条目 | 行业、公司、技术、商业模式、财务预测等解释结构的版本化条目 |
| 规则工件 | 清洗、抽取、对账、仲裁、准入、派生、课题、复盘等操作规则 |
| 驱动树 | 将公司业务和财务科目连接到预测节点的结构 |
| 假设账本 | 记录模型当前采用的输入假设及其来源、置信度、冲突状态 |
| 影子快照 | 系统自动生成、用于观察和比较的非正式预测快照 |
| 正式快照 | 通过发布闸门、进入正式输出的预测快照 |
| 调度脊柱 | 感知、触发、准入、行动、队列的运行控制结构 |
| 课题 | 绑定目标变量、主动寻找证据和更新模型的复合行动 |
| 冷存池 | 暂时不足以采信但可能累计触发的弱信号集合 |
| M-LOG | 执行日志，记录系统每次行动的输入、输出、版本和结果 |
| as-of 复盘 | 回到某一历史时点，只用当时可得信息评价当时预测 |

# 18. 一句话收口

这个系统不是“更强的搜索助手”，而是一个围绕财务预测长期运转的证据、规则、假设和复盘机器。

v0.6 的关键收口是把原来混在 L3 的证据系统拆清楚：

```text
L2 观测证据层：来源说了什么，当前采信什么，哪里冲突；
L3 派生证据层：系统基于证据和规则构造出了什么。
```

L1 可以现实地轻一些：公告 PDF 自维护，其他来源保留访问与追溯记录。真正不能轻的是 L2 观测证据、L3 派生证据、L5 预测快照、L6 调度脊柱和财报复盘。只要进入预测，信息就必须能被解释、被追溯、被冲突、被重算、被复盘。
