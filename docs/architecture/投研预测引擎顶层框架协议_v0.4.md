# AI Agent 投研预测引擎顶层框架协议 v0.5

> 目标：一个人机交互下的、数据驱动的、持续运行的财务预测引擎。  
> 当前正式产出：契约口径下的公司未来季度 / 年度净利润预测。  
> 未来演进方向：从净利润预测，逐步扩展到利润表、资产负债表、现金流量表的完整预测。

---

## 版本说明

v0.5 基于 v0.4 继续收口，核心调整是重新定义第一层。

上一版容易隐含一个假设：系统自己长期维护大部分 raw data，并以 append-only 的方式保存所有原始资料。这个假设不符合当前工程现实。更合理的设定是：

1. **上市公司公告 / 财报 PDF 是自维护资料锚**。这是系统最重要的披露锚，必须尽量自己维护原文、版本、文本抽取结果和定位能力。
2. **其他大部分资料不强求自维护 raw data**。包括购买的本地数据服务、券商 API、MCP、web search、公司官网、论文 API、行业数据库等。系统更多维护的是访问记录、来源标识、查询参数、返回摘要、证据引用和必要缓存。
3. **L2 以后必须保留可追溯记录，但不追求 100% 原文级追溯**。系统接受“追溯等级”差异：有些证据能追到 PDF 页码，有些只能追到 API 查询和访问时间，有些只能追到人工记录。追溯等级会影响置信度和使用边界。
4. **证据账本仍然是系统核心**。L1 可以轻，但 L2/L3 不能虚。只要某条信息进入预测，它就必须在证据账本、假设账本、工作底稿、快照和执行日志里留下可解释路径。

这版文档不写代码、不写 SQL、不写详细 schema、不写目录结构。它只锁定顶层架构、层间职责、运行机制和硬边界。

# 0. 系统宪法

## 0.1 系统本质

这不是问答系统，不是单次 case 驱动的研究助手，也不是“人提问、机器搜索、返回答案”的工具。

它是：

> **一个人机交互下的、数据驱动的、持续运行的财务预测引擎。**

系统长期吸收资料，沉淀证据，维护公司预测模型，发现冲突，触发研究课题，更新假设，生成预测快照，并在财报季复盘自己的错误。

## 0.2 唯一正式产出

系统唯一正式产出是：

> **公司未来财务预测。**

当前阶段对外输出：

> **契约口径下的未来季度 / 年度净利润预测。**

“契约口径”可以很轻：

- v0：原始财报口径；
- v0.1：原始财报口径 + 扣非；
- 后续：逐步加入公司特定调整项、一次性项目、会计政策变化处理、业务重组口径等。

## 0.3 系统不做什么

系统不做：

- 估值；
- 选股；
- 买卖建议；
- 判断便宜与否；
- 组合构建；
- 市场短期博弈判断。

这些由投资经理拿着预测结果另行判断，不进入本系统边界。

## 0.4 设计原则

1. **预测优先**：任何模块存在的理由，都必须能解释它如何改善财务预测，或改善预测的可解释性、可复盘性。
2. **证据优先**：系统不直接相信观点。观点必须被拆成 claim，进入证据账本，被标注来源、口径、时间、置信度和冲突状态。
3. **披露为锚**：财报、公告、业绩预告、问询函回复是硬锚；外部细分数据可以补充披露锚，但不能无解释地违反披露锚。
4. **L1 可轻，L3 必须重**：原始资料层不强求全量 raw 归档，但证据账本、假设账本、快照和日志必须可解释。
5. **推拉双向**：既有数据流入后自动传播的“推”，也有从预测问题出发主动找资料的“拉”。
6. **课题不是旁支**：研究课题是调度行动的一类，不是第二套系统。课题拿到的资料仍然必须走入库处理带和证据账本。
7. **规则与调度分离**：规则是乐谱，调度是指挥，行动是演奏。调度不承载研究方法本身。
8. **人批关键认知变化**：框架、口径契约、重大冲突裁决、正式预测发布必须有人参与。
9. **LLM 不做确定性工作**：能用确定性规则做的，不交给 LLM。LLM 主要负责抽取、归纳、消歧、提出候选解释和生成待办。
10. **快照不可回改**：预测快照发布后不可修改，只能产生新快照或重述复盘。

# 1. 总体形态

系统由一条主管道、一个旁路层、一根调度脊柱、两条记录线和一个拉式课题回路组成。

```text
                        ┌──────────────────────┐
                        │  L4 知识与规则层       │
                        │  框架架 / 规则工件架   │
                        └──────────┬───────────┘
                                   │ 加载规则 / 引用框架
                                   ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ L1 资料接入层 │ -> │ L2 入库处理带 │ -> │ L3 证据账本   │ -> │ L5 个股预测层 │ -> │ 预测快照      │
│ 披露锚归档    │    │ 抽取/校验/分流│    │ claims/current│    │ 驱动树/假设   │    │ 正式/影子     │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
        ▲                    ▲                  ▲                    │
        │                    │                  │                    │
        │                    └────── 课题主动获取资料 ────────────────┘
        │
        └──────────── web / API / MCP / 券商库 / 官网 / 人工上传 / 公告PDF服务

               ┌──────────────────────────────────────────────┐
               │ L6 调度脊柱：感知 / 触发 / 准入 / 行动 / 队列 │
               └──────────────────────────────────────────────┘

               ┌────────────────────┐      ┌────────────────────┐
               │ 执行日志 M-LOG      │      │ 人机待办            │
               │ 输入/输出/版本/动作 │      │ 裁决/调研/补证据     │
               └────────────────────┘      └────────────────────┘
```

## 1.1 主管道：数据推动预测

主管道是：

```text
资料接入与披露锚归档
→ 入库处理带
→ 证据账本
→ 个股预测层
→ 预测快照
```

当新公告、新财报、新纪要、新研报、新 API 数据、新网页资料进入时，系统不是直接改预测，而是先转成证据账本里的 claim，再由调度脊柱判断是否影响假设和预测。

## 1.2 拉式回路：预测问题反向找资料

系统不能只等待数据流入。新增个股、模型冷启动、关键假设缺失、冲突无法裁决时，必须从预测问题反向发起研究课题：

```text
预测变量 / 阻塞问题
→ 研究课题
→ 查证据账本和冷存池
→ 主动获取资料
→ 入库处理带
→ 证据账本
→ 更新假设 / 驱动树 / 框架条目
→ 重算预测
```

例如，中际旭创某类产品毛利率不清楚，不是让 agent 直接回答“毛利率是多少”，而是立一个绑定目标变量的课题：

```text
目标变量：中际旭创某产品线 2026Q2 / 2026E 毛利率假设
信息需求：产品价格、成本结构、良率、工艺、原材料、客户结构、产能利用率、竞争格局
资料路径：先查账本 → 查冷存 → 主动检索 / API / 纪要 / 调研 → 入库 → 形成 claims → 更新假设
```

课题不是第二套系统，它只是调度行动库里的一类复合行动。

## 1.3 旁路层：知识与规则

知识与规则层不直接装事实。它装两类东西：

1. **框架架**：怎么理解一个行业、公司、业务、技术路线、商业模式、财务结构。
2. **规则工件架**：怎么抽取、清洗、对账、仲裁、建模、复盘、设定口径契约。

它被主管道和课题回路加载使用，但不替代证据账本。

## 1.4 调度脊柱

调度脊柱负责五件事：

```text
感知 → 触发 → 准入 → 行动 → 队列
```

- 感知：知道数据、claim、假设、框架、契约、快照之间的大致依赖关系。
- 触发：识别新资料、新 claim、状态变化、人工指令、财报披露、模型陈旧等事件。
- 准入：判断是否值得执行、立即执行、低优先入队、仅记录或转人工。
- 行动：执行抽取、校验、对账、检索、重算、生成待办、课题推进等动作。
- 队列：管理失效、重算、低优先任务、人工待办和失败重试。

# 2. L1：资料接入与披露锚归档层

## 2.1 重新定义 L1

L1 不再定义为“系统自有 raw data 仓库”。

更准确的定义是：

> **资料访问、来源登记、披露锚归档和检索记录层。**

它负责让系统知道资料来自哪里、如何取得、何时取得、是否有原文、是否可复查、是否允许缓存、是否适合作为预测证据。

## 2.2 L1 自己维护什么

当前阶段，系统应该优先自己维护：

### 2.2.1 上市公司公告 / 财报 PDF

这是最重要的自维护资产，包括：

- 年报；
- 半年报；
- 季报；
- 业绩预告；
- 业绩快报；
- 正式公告；
- 问询函及回复；
- 交易所监管函件；
- 重大资产重组、股权激励、定增、回购、分红等专项公告。

这些资料是披露锚，原则上应该保存：

- 原始 PDF；
- 来源渠道；
- 下载时间；
- 公告日期；
- 公告标题；
- 公司代码；
- 公告编号；
- 文件哈希；
- 文本抽取版本；
- 表格抽取版本；
- 页码 / 段落 / 表格定位。

这不是为了形式主义，而是为了后面能回答：

> 某个净利润预测里的某个假设，最底层到底锚在哪份公告、哪一页、哪个表、哪个披露口径？

### 2.2.2 人工上传资料

包括：

- 路演笔记；
- 调研纪要；
- 专家访谈纪要；
- 微信群关键内容归档；
- 自己整理的公司跟踪笔记；
- PPT、Excel、图片、录音转写文本。

人工资料不是天然可信，但它们通常很有价值。系统必须保存其来源上下文：

- 谁提供；
- 何时提供；
- 是否一手；
- 是否转述；
- 是否可复核；
- 是否有原始录音 / 截图 / 会议链接；
- 是否涉及合规限制。

## 2.3 L1 不强求维护什么

以下资料在当前阶段不强求自维护全量 raw data：

- 购买的本地数据服务；
- 券商数据库 API；
- Wind / iFinD / Choice 等结构化数据服务；
- MCP 工具返回结果；
- web search 结果；
- 公司官网网页；
- 政府、协会、海关、行业数据库网页；
- 论文 API；
- 新闻站点；
- 自媒体、大 V、论坛内容。

这类资料可以只保存：

- 数据源名称；
- 查询参数；
- 返回时间；
- 返回字段摘要；
- URL 或结果 ID；
- 访问工具；
- 关键片段；
- 是否可重查；
- 是否允许缓存；
- 被抽取出的 claims；
- L2/L3 的证据引用关系。

原则是：

> L1 不追求“我拥有一切原文”，但必须记录“我为什么认为这条证据曾经存在、来自哪里、什么时候看到、以什么形式进入系统”。

## 2.4 来源分级

来源分级不是为了机械打分，而是为了决定：能否采信、能否单独影响预测、是否需要对账、是否只能进弱信号池。

| 层级 | 类型 | 例子 | 使用原则 |
|---|---|---|---|
| Tier 0A | 硬披露锚 | 财报、公告、业绩预告、业绩快报、问询函回复 | 最高优先级；原文必存；用于对账和复盘 |
| Tier 0B | 软披露锚 | 投关活动记录、业绩说明会、官网材料、公司公开路演 | 可提升置信度；仍需结合硬披露锚 |
| Tier 1 | 结构化外部数据 | 券商库、政府统计、协会、海关、产业数据库 | 可作为重要输入；需记录口径和查询时间 |
| Tier 2 | 外部研究与观点 | 研报、纪要、专家访谈、权威媒体深度报道 | 可形成估算和观点型 claims；需对账、冲突检测 |
| Tier 3 | 弱公开信源 | 自媒体、评论员、大 V、论坛、社群传闻 | 默认进弱信号池；不能单独成为正式预测采信依据 |
| Tier F | 自有一手资料 | 自己调研、专家交流、路演笔记 | 高价值但不天然高置信；看可复核性和披露锚一致性 |

## 2.5 追溯等级

因为 L1 不追求全量 raw，自然要接受不同追溯等级。

| 追溯等级 | 含义 | 例子 | 对置信度的影响 |
|---|---|---|---|
| G0 | 原文级追溯 | PDF 原文 + 页码 + 段落 / 表格位置 | 最高；可作为硬锚 |
| G1 | 数据服务级追溯 | 数据源 + 查询参数 + 返回字段 + 时间 | 可正式使用，但要注意供应商口径 |
| G2 | 网页 / 工具级追溯 | URL / 工具名 + 访问时间 + 关键片段 | 可辅助使用；重要结论需交叉验证 |
| G3 | 人工记录级追溯 | 会议笔记、访谈纪要、转述记录 | 可用但需标记不确定性 |
| G4 | 弱追溯 | 来源模糊、截图不完整、二手传闻 | 原则上只进冷存或待核验，不直接影响正式预测 |

硬规则：

- 追溯等级必须进入 claim 元数据；
- 弱追溯 claim 必须有置信度上限；
- G4 不能单独支撑正式预测；
- 如果弱追溯信息很重要，应触发课题或人工待办，而不是直接采信。

## 2.6 检索记录也是数据

主动 search、API 查询、MCP 调用即使没有找到结果，也要记录。

“查过但没找到”有三个价值：

1. 避免重复检索；
2. 说明某个结论目前缺证据，而不是系统忘了查；
3. 可转为人机待办：例如“公开渠道未找到某产品毛利率，需要问卖方 / 专家 / 公司”。

# 3. L2：入库处理带

## 3.1 L2 的定位

L2 是系统最重的工程区之一。它不按“文件”处理，而按“信息单元”处理。

一篇纪要里可能同时有：

- 一个高价值订单数据；
- 一个低可信卖方观点；
- 一段行业背景；
- 一个和已有资料冲突的价格信息；
- 一句无法核验的市场传闻；
- 一个能触发新课题的问题。

L2 的职责不是判断最终真相，而是把混合资料拆成可入账、可冲突、可对账、可追溯、可触发的结构化证据候选。

## 3.2 L2 的标准处理链

标准处理链：

```text
资料登记
→ 轻处理
→ 信息单元切分
→ 主体 / 时间 / 指标 / 事件识别
→ claim 抽取
→ 口径标注
→ 来源与追溯等级标注
→ 校验隔离
→ 去重与新旧判断
→ 披露锚对账
→ 冲突检测
→ 置信度初评
→ 重要性评分
→ 分流
```

## 3.3 轻处理

轻处理优先用确定性方法，包括：

- 去重；
- 文件类型识别；
- 时间识别；
- 公司代码 / 名称匹配；
- 证券简称消歧；
- 资料类型分类；
- 是否公告 / 财报 / 纪要 / 研报 / 访谈 / 新闻 / 社媒；
- 语种识别；
- 基础文本抽取；
- 表格初步抽取。

LLM 只在模糊文本、复杂切分、主体消歧、非结构化摘要、长文本信息抽取时介入。

## 3.4 claim 抽取

claim 是证据账本的基本单位。L2 要把资料拆成 claim 候选，例如：

- 某公司某产品收入是多少；
- 某产品价格环比上涨；
- 某客户订单开始放量；
- 某产线良率改善；
- 某公司某项费用率下降；
- 某供应商与公司存在供应关系；
- 某技术路线可能替代另一技术路线；
- 某研究员预测明年净利润多少。

抽取时必须区分：

- 事实；
- 观点；
- 估算；
- 预测；
- 传闻；
- 关系；
- 解释；
- 假设。

不同性质进入账本后的采信规则不同。

## 3.5 校验隔离

生产 agent 和校验 agent 必须隔离。

- 生产 agent：负责从原文抽取 claim、给出初步结构化结果。
- 校验 agent：只看原文和抽取结果，不看生产 agent 的推理过程。
- 冲突 agent：对比历史 claims、当前采信 claims、披露锚和相关假设。
- 重要性 agent：可以提出重要性建议，但最终是否执行由确定性准入规则决定。

目的不是迷信多 agent，而是降低单次抽取幻觉和顺着一个错误推理继续错下去的概率。

## 3.6 披露锚对账

外部细分数据不能无解释地违反披露锚。

例如，卖方给出某公司分产品收入和毛利，如果分项加总与财报披露的业务收入差距过大，系统不能直接采信。它应当：

1. 识别可能口径差异；
2. 查找是否存在合并范围、期间、币种、税费、一次性项目、产品分类差异；
3. 按对账规则计算偏离程度；
4. 根据偏离程度调整置信度；
5. 超阈值转 contested；
6. 必要时生成待办或情景分支。

对账规则本身不放在 L2，它放在 L4 的规则工件架。L2 只是加载规则并执行。

## 3.7 分流结果

L2 的输出有四类：

1. **进入证据账本**：可能影响预测，或者有长期复用价值。
2. **进入冷存 / 弱信号池**：单条不足以行动，但值得累计观察。
3. **进入人机待办**：需要人工裁决、补证据、问卖方、问专家、查公告。
4. **丢弃 / 垃圾池**：低质量、重复、无法定位主体、没有预测价值、明显噪声。

丢弃也要可粗略审计。至少要能知道某批资料为什么没有进入证据账本。

---

# 4. L3：证据账本

## 4.1 L3 的定位

证据账本是系统最核心的长期资产。

L1 可以不保存所有 raw，L2 可以不断改抽取方法，L4 可以不断演进规则，但 L3 必须长期稳定地保存：

- 系统见过哪些关于世界的陈述；
- 这些陈述来自哪里；
- 它们是什么性质；
- 当前是否被采信；
- 是否有冲突；
- 是否影响过预测；
- 后来是否被财报证伪。

## 4.2 claim 的基本内容

一个 claim 至少要表达清楚：

- 主体：公司、子公司、产品、客户、供应商、行业、区域；
- 指标 / 事件：收入、毛利率、价格、订单、产能、良率、费用率、供应关系等；
- 陈述内容：数值或文本；
- 单位：元、美元、吨、片、台、bps、百分比等；
- 时间范围：某季度、某年度、某时点、未来期间；
- 口径标签：财报口径、扣非口径、产品口径、区域口径、含税 / 不含税、合并 / 单体等；
- 性质：事实、估算、观点、预测、传闻、关系；
- 来源链：来源类型、追溯等级、访问记录、原文定位；
- 来源层级：Tier 0A / 0B / 1 / 2 / 3 / F；
- 置信度；
- 冲突状态；
- 采信状态；
- 下游引用关系。

这不是数据库 schema，只是架构层必须保留的信息维度。

## 4.3 三类 claim

### 4.3.1 原始 claims

原始 claims 是所有来源的并列陈述。它们不互相覆盖。

例如：

- A 研报认为某产品 2026 年毛利率 35%；
- B 纪要认为毛利率 30% 左右；
- 公司公告没有披露单品毛利率，但披露整体毛利率 28%；
- 专家访谈认为新产品初期良率较低。

这些都应该并列进入账本，而不是只保留一个“最终答案”。

### 4.3.2 当前采信 claims

当前采信 claims 是系统在某个时点认为最适合进入预测的陈述或数值。

它不是第二套真相，而是对原始 claims 经过规则、对账、人工裁决后的当前视图。

### 4.3.3 derived claims

derived claims 是由原始 claims 经过口径契约、计算规则或派生逻辑生成的结果。

例如：

- 从原始财报口径净利润派生扣非净利润；
- 从分业务收入和成本派生毛利率；
- 从供应关系 claims 和产品分类模板派生某产品供应链视图。

derived claims 逻辑上仍属于证据账本，不允许变成第二套独立真相。

## 4.4 claim 状态机

状态机必须存在，但枚举可以后续微调。当前建议：

| 状态 | 含义 |
|---|---|
| pending | 已入账，尚未采信或裁决 |
| accepted | 当前被采信，可进入假设或预测 |
| contested | 与其他证据、披露锚或模型假设冲突 |
| rejected | 已被否定或不再适合采信 |
| archived | 历史保留，不再参与当前预测 |
| provisional | 临时采信，等待补证据或财报验证 |
| scenario-set | 不确定性过大，进入情景分支而非单点采信 |

高重要性 contested claim 不能被静默吞掉。它必须至少触发：

- 人工待办；
- 情景分支；
- 影子预测；
- 或正式预测阻塞。

## 4.5 关系型 claims

供应商、客户、产业链归属、竞争关系、份额、技术路线位置，不应放在知识层作为静态事实。

它们应该作为关系型 claims 进入证据账本。

原因很简单：这些关系会变，有来源，会冲突，会影响预测。

知识层只放结构模板，例如“光模块行业应该看客户结构、速率迭代、良率、产能、价格、DSP/光芯片/封装/测试环节”。至于“某家公司当前对应哪个客户、哪个产品、哪个供应商”，应由账本中的关系 claims 动态生成。

## 4.6 按时点查询

账本必须支持 as-of 语义：

> 回到某个历史时点，系统当时知道什么、采信什么、忽略什么、冲突什么？

没有 as-of，就无法做真正复盘。因为你不能用今天才知道的信息去评价昨天的预测。

# 5. L4：知识与规则层

## 5.1 L4 的定位

L4 是旁路层，不是主管道。

它不装“世界事实”，而装：

1. **如何理解世界的框架**；
2. **如何操作资料、证据和预测的规则**。

L4 被 L2、L3、L5、L6 和课题行动加载使用。

## 5.2 框架架

框架架存放解释结构，例如：

- 行业研究框架；
- 产业链结构模板；
- 行业驱动因子模板树；
- 商业模式模板；
- 技术路线框架；
- 政策解读方法；
- 公司治理和管理层动机透镜；
- 财务预测模板；
- 三大表建模框架；
- 现金质量检查框架；
- 常见利润桥模板。

框架条目必须：

- 条目化；
- 版本化；
- 可引用；
- 可被快照记录；
- 可被复盘归因；
- 重大修改需要人批。

## 5.3 规则工件架

规则工件架存放操作规则，例如：

- 清洗规则；
- 抽取规则；
- 对账规则；
- 冲突仲裁原则；
- 置信度规则；
- 重要性准入规则；
- 口径契约；
- 课题模板；
- 行动手册；
- 财报复盘归因规则；
- 正式快照发布规则。

规则和调度必须分离。

- 规则回答：应该怎么判断、怎么处理、怎么对账。
- 调度回答：什么时候执行、执行什么、由谁执行、执行后放到哪里。
- 行动回答：实际执行一次抽取、校验、重算、检索、课题推进。

## 5.4 口径契约

口径契约归入规则工件架，但它是最高优先级规则之一。

它定义：

- 历史财务数据如何重构；
- 未来预测按什么口径输出；
- 财报季复盘按什么口径对比；
- 哪些调整进入净利润；
- 哪些调整只作为备注；
- 哪些项目需要公司附录。

口径契约采用：

```text
全局基准 + 公司附录
```

全局基准可以很轻：

- v0：原始财报口径；
- v0.1：原始财报口径 + 扣非；
- v0.2：增加常见一次性项目处理规则；
- 后续：公司特定附录。

每条契约政策最好拆成两段：

1. 文字理由：为什么这么处理；
2. 可执行规则：系统如何实际处理。

引擎只执行规则段，文字理由供人和 agent 理解。

硬规则：

> 历史重构、未来预测、财报复盘必须引用同一版口径契约。

如果契约升版，应触发相关公司历史重算、模型重估和复盘重述，但不能覆盖原预测快照。

## 5.5 可训练大脑

你希望系统不是静态规则堆，而是一个能通过复盘不断变好的“投资大脑”。这个想法在架构上落在 L4。

但它不是一开始就训练大模型，而是先建立可学习结构：

```text
框架条目 / 规则工件
→ 被预测模型引用
→ 产生预测快照
→ 财报验证
→ 误差归因到条目
→ 形成 diff 提案
→ 历史重放评测
→ 人批晋级
```

类比深度学习：

- 前向传播：框架 + 规则 + 证据 → 假设 → 预测；
- loss：预测与财报实际之间的误差；
- 反向归因：定位是哪个来源、规则、框架条目、假设或人工覆盖导致误差；
- 参数更新：修改框架条目、规则工件、契约附录、预测模板；
- 冻结层：系统边界、证据账本、快照不可回改、调度原则；
- 可训练层：行业模板、规则工件、公司附录、托底规则、课题模板。

早期训练不是自动化模型训练，而是“版本化文本 + 人审 + 历史重放”。数据积累后，再考虑自动评测、模板推荐、甚至微调。

## 5.6 共享与公司特例

每家公司每年只有 4 个财报验证点。逐家公司学习样本太少。

因此，行业共性必须尽量进入共享模板条目，公司特例才进入公司附录。

例如：

- “光模块公司收入预测需要拆客户、速率、产品代际、价格、产能、良率”应进入行业共享模板；
- “某公司某客户供货节奏特殊”应进入公司模型或公司附录；
- “某行业玩家名单”不进框架层，应由关系型 claims 派生。

# 6. L5：个股预测层

## 6.1 L5 的定位

个股预测层负责把证据、规则、框架和人工判断转化为公司未来财务预测。

当前阶段聚焦净利润，但内部不能只存一个净利润数字。必须保留利润桥、关键假设、现金质量检查和中间节点。

## 6.2 每家公司一个预测对象

每家公司对应一个预测对象，内部至少包括：

1. 驱动树；
2. 利润桥；
3. 假设账本；
4. 数据绑定与订阅；
5. 现金质量检查；
6. 情景分支；
7. 工作底稿；
8. 正式预测快照；
9. 影子预测快照；
10. 财报季复盘记录。

## 6.3 驱动树

驱动树是公司财务预测的结构。

净利润阶段也不能只预测净利润。至少要有完整利润桥：

```text
收入
→ 成本
→ 毛利
→ 费用
→ 经营利润
→ 利润总额
→ 所得税
→ 少数股东损益
→ 归母净利润
```

每个节点都要知道：

- 这个节点来自哪个框架条目；
- 用了哪些 claims；
- 用了哪些假设；
- 是否是托底规则；
- 是否有冲突；
- 是否有人工覆盖；
- 下次验证点是什么。

## 6.4 托底规则

早期不可能每个科目都研究到位。因此允许托底规则，例如：

- 历史均值；
- 最近趋势；
- 行业均值；
- 管理层指引；
- 卖方一致估算；
- 人工临时假设。

但托底规则必须显式标记，不能伪装成深度研究。

系统要能回答：

> 这次净利润预测里，哪些节点是研究化 driver，哪些节点只是托底？

长期演进方向是逐步把高影响托底节点替换为研究化 driver。

## 6.5 假设账本

假设账本不是证据账本。

证据账本记录世界陈述；假设账本记录模型当前采用的输入。

一个假设至少要说明：

- 对应公司；
- 对应业务 / 产品 / 客户 / 区域；
- 对应指标；
- 对应期间；
- 假设值；
- 来源 claims；
- 使用的口径契约；
- 使用的框架条目；
- 是否人工覆盖；
- 置信度；
- 冲突状态；
- 下次验证点；
- 对净利润敏感度。

## 6.6 数据绑定与订阅

预测节点必须注册自己关心什么数据。

绑定键可以概念化为：

```text
主体 × 业务/产品/客户/区域 × 指标/事件 × 期间 × 口径 × 来源优先级
```

当证据账本里出现匹配的 claim，调度脊柱才能知道它可能影响哪个假设、哪个节点、哪个公司快照。

## 6.7 现金质量检查

即使当前正式输出只是净利润，也必须保留现金质量检查。

原因是净利润容易被收入确认、应收、存货、合同负债、资本化、减值、一次性项目污染。

现金质量检查至少包括：

- 应收账款；
- 存货；
- 合同负债；
- 经营现金流；
- 资本开支；
- 折旧摊销；
- 资产减值；
- 费用资本化；
- 非经常性损益。

它不一定一开始影响净利润点值，但必须影响置信度、污染标记、待办优先级和复盘归因。

## 6.8 情景分支

情景分支只用于高重要性 contested driver。

例如：

- 某产品毛利率有两个可信来源但差异很大；
- 某大客户订单是否放量不确定；
- 某产品价格是报价还是成交价不清楚；
- 某产能是否按期释放存在分歧。

情景分支不能无限扩散。必须限制：

- 分支数量；
- 分支变量数量；
- 分支持续时间；
- 是否影响正式预测；
- 是否需要人工裁决。

主预测使用当前采信假设；情景分支用于预测区间、敏感性和待办排序。

## 6.9 正式快照与影子快照

- **影子快照**：系统可以自动生成，用于观察、比较、提前发现变化。
- **正式快照**：进入对外正式输出，需要通过发布闸门。

正式快照必须保存：

- 证据版本；
- 口径契约版本；
- 框架条目引用；
- 规则工件版本；
- 模型版本；
- 假设版本；
- 中间计算节点；
- 利润桥；
- 现金质量检查；
- 情景分支；
- 敏感变量；
- 最脆弱假设；
- 人工覆盖记录；
- 发布理由；
- 未解决冲突。

发布后不可回改。

# 7. L6：调度脊柱

## 7.1 定义

调度脊柱不是研究方法本身。它负责让系统持续运行。

结构是：

```text
感知 → 触发 → 准入 → 行动 → 队列
```

## 7.2 感知：最小血缘图 + 订阅索引

第一阶段不要把血缘图做得过重。目标不是完整知识图谱，而是回答四个工程问题：

1. 某条 claim 变化，会影响哪些假设？
2. 某个口径契约升版，会影响哪些 derived claims 和预测公式？
3. 某个框架条目变化，会影响哪些公司模型和假设？
4. 某个假设变化，会影响哪些预测节点和快照？

因此最小血缘边可以先锁定：

```text
source_ref / source_event → claim
claim → assumption
contract → derived claim / formula
framework_entry / rule_artifact → driver_tree / assumption / action
assumption → forecast_node
forecast_node → snapshot
research_topic → produced claims / assumptions / framework diffs / rule diffs
```

其他关系先通过执行日志保留，不急着建复杂图。

## 7.3 触发

触发来源包括：

### 7.3.1 事件触发

- 新公告；
- 新财报；
- 新研报；
- 新纪要；
- 新 API 数据；
- 新 web 检索结果；
- 新 claim；
- claim 状态变化；
- 人工裁决；
- 口径契约升版；
- 框架条目变更；
- 规则工件变更；
- 财报披露。

### 7.3.2 定时触发

- 冷存池扫描；
- 弱信号累计扫描；
- 假设过期检查；
- 模型陈旧检查；
- 临近财报检查；
- 失败任务重试；
- 影子快照定期重算。

### 7.3.3 人工触发

人工触发是一等触发源，例如：

- 新增个股研究；
- 开启某个产品毛利率课题；
- 要求重算某公司预测；
- 强制查看某条冲突；
- 手动裁决一个假设；
- 让系统生成调研待办。

人工触发可以直通，但仍要经过准入记录：系统需要知道这件事为什么被执行。

## 7.4 准入

准入不是一个复杂“智能体”，而是触发和行动之间的策略层。

作用是：

1. 控成本；
2. 控噪声；
3. 控人的注意力；
4. 保证事后可审计。

候选行动根据重要性映射到：

- 立即执行；
- 低优先入队；
- 仅记录；
- 转人工确认；
- 丢弃。

重要性特征包括：

- 来源 Tier；
- 追溯等级；
- 影响公司数量；
- 涉及预测期间；
- 与当前假设偏离程度；
- 对净利润敏感度；
- 是否临近财报；
- 是否涉及高置信冲突；
- 是否来自硬披露锚；
- 是否解决已有待办。

LLM 可以对定性材料打辅助分，但不能单独决定准入结果。

## 7.5 行动库

行动分为原子行动和复合行动。

### 7.5.1 原子行动

- 获取资料；
- 登记来源；
- 归档公告 PDF；
- 文本抽取；
- 表格抽取；
- claim 抽取；
- 校验；
- 对账；
- 冲突检测；
- 更新 claim 状态；
- 更新假设；
- 重算预测节点；
- 生成影子快照；
- 生成正式快照候选；
- 生成待办；
- 冷存归档；
- 检索未得记录。

### 7.5.2 复合行动

- 入库流程；
- 变更传播；
- 研究课题；
- 个股初始化；
- 财报季复盘；
- 口径契约升版；
- 框架条目重放评测；
- 来源可信度更新。

每个行动必须声明：

- 输入；
- 输出；
- 加载的规则工件；
- 使用的框架条目；
- 执行者：代码 / agent / 人；
- 日志记录内容；
- 失败后怎么处理。

## 7.6 失效队列

当上游变化后，系统不全量重算，而是：

```text
定位受影响对象
→ 判断重要性
→ 标记失效
→ 入失效队列
→ 自动重算 / 影子重算 / 人工确认 / 仅观察
→ 生成新快照或待办
```

失效不是错误，而是系统正常运行状态。

# 8. 研究课题：拉式回路

## 8.1 为什么必须有课题

成熟公司模型可以依赖数据流入后持续更新。

但新增个股、冷启动、深度 driver 建设、冲突裁决，不能只靠被动信息流。很多研究是从预测问题出发，一层层追下去：

```text
净利润预测不稳
→ 毛利率假设不稳
→ 某产品毛利不清楚
→ 成本结构不清楚
→ 原材料价格、良率、工艺、客户结构不清楚
→ 需要主动找资料 / 问专家 / 查公告 / 查竞品
```

这就是课题。

## 8.2 课题定义

课题是复合行动，必须绑定目标变量。

一个课题至少包含：

- 目标公司 / 行业；
- 目标变量；
- 阻塞问题；
- 预测影响；
- 当前证据缺口；
- 信息需求清单；
- 可用来源路径；
- 子课题；
- 预算限制；
- 预期产出；
- 结案标准。

禁止无边界课题，例如“研究一下 AI 光模块”。

允许的写法是：

> 为了更新中际旭创 2026E 毛利率假设，研究 800G / 1.6T 产品价格、良率、客户结构和产能释放对产品毛利的影响。

## 8.3 课题来源

课题可以来自：

- 人工主动提出；
- 新增个股初始化；
- 模型发现关键假设缺失；
- 高重要性 claim 冲突；
- 弱信号池累计；
- 财报复盘发现误差来源；
- 框架条目缺失；
- 数据流自动提出候选，等待人批准。

## 8.4 课题执行循环

```text
立题
→ 生成信息需求清单
→ 查证据账本
→ 查冷存池
→ 主动获取资料
→ 新资料进入 L2
→ 形成 claims
→ 对账和冲突检测
→ 更新假设 / 驱动树 / 框架 diff / 规则 diff
→ 必要时开子课题
→ 结案或转人工待办
```

关键硬规则：

> 课题主动获取的资料，必须和被动资料一样进入 L2 和 L3，不能直接绕过证据账本改预测。

## 8.5 课题预算

课题必须控制递归和成本。

需要限制：

- 最大子课题深度；
- 最大子课题数量；
- 最大检索轮次；
- 最大 agent 执行次数；
- 最大人工待办数量；
- 是否允许影响正式预测；
- 结案时间或复查时间。

否则系统会无限研究，无法产出预测。

## 8.6 新增个股初始化

新增个股是最大的标准课题。

初始化目标不是“一次性研究完”，而是形成：

> **能出影子预测的最小公司模型。**

初始化课题至少覆盖：

1. 公司主营业务；
2. 产品 / 服务拆分；
3. 收入结构：产品、客户、区域；
4. 成本结构；
5. 毛利率拆分；
6. 费用结构；
7. 上下游关系；
8. 供应商与客户；
9. 产能与扩产；
10. 竞争格局；
11. 技术路线；
12. 价格驱动；
13. 历史财务重构；
14. 现金质量；
15. 管理层诉求和激励；
16. 主要争议点；
17. 最小预测驱动树；
18. 未解决待办。

未研究透的部分允许托底，但要显式标记。

# 9. 标准流程

## 9.1 入库流程

```text
资料进入 / 访问记录生成
→ L1 登记来源与追溯等级
→ L2 轻处理
→ L2 信息单元切分
→ L2 claim 抽取
→ L2 校验隔离
→ L2 对账和冲突检测
→ L2 重要性评分
→ 分流到 L3 / 冷存 / 待办 / 丢弃
→ L6 判断是否触发下游行动
```

## 9.2 变更传播流程

```text
claim / 假设 / 契约 / 框架 / 规则 / 人工裁决发生变化
→ 调度脊柱查订阅与最小血缘
→ 定位受影响假设、节点、公司、快照
→ 准入判断
→ 入失效队列
→ 重算影子预测或生成待办
→ 必要时生成正式快照候选
```

## 9.3 主动课题流程

```text
预测问题或人工需求
→ 立题
→ 查账本和冷存
→ 主动获取资料
→ 新资料标准入库
→ 更新 claims 和假设
→ 重算预测
→ 结案记录进入 M-LOG
```

## 9.4 新增个股流程

```text
人工指定公司
→ 启动个股初始化课题
→ 建立最小公司档案
→ 建立初始驱动树
→ 接入公告 PDF 和历史财报
→ 形成历史利润桥
→ 形成当前核心假设
→ 标记托底节点和缺口
→ 生成影子预测
→ 形成待办列表
```

## 9.5 财报季复盘流程

```text
财报 / 公告入库
→ 作为硬披露锚进入账本
→ 按快照当时口径契约重构实际值
→ 对比正式快照和影子快照
→ 按利润桥逐节点定位偏离
→ 做误差归因
→ 回写来源可信度、规则、框架、假设、待办
→ 必要时提出框架 / 契约 / 规则 diff
```

复盘不能只看净利润误差。必须定位：

- 收入错了；
- 毛利率错了；
- 费用率错了；
- 税率错了；
- 少数股东损益错了；
- 一次性项目错了；
- 口径错了；
- 披露理解错了；
- 数据源错了；
- 抽取错了；
- 仲裁错了；
- 框架结构错了；
- 人工覆盖错了；
- 蒙对了但逻辑错了。

## 9.6 口径契约升版流程

```text
发现口径问题
→ 提出契约 diff
→ 人工审核
→ 小范围历史重算
→ 检查影响公司和预测
→ 契约升版
→ 受影响 derived claims 和预测公式失效
→ 生成新快照或重述复盘
```

契约升版不能悄悄改变历史快照。

## 9.7 框架条目演进流程

```text
财报误差 / 课题发现 / 人工判断
→ 定位框架条目问题
→ 提出框架 diff
→ as-of 历史重放评测
→ 人工批准
→ 框架条目升版
→ 受影响公司模型重检
```

# 10. 冷存 / 弱信号池

## 10.1 定位

冷存池收纳暂时不足以进入证据账本或不足以影响预测的信息。

典型信息：

- 单条社媒传闻；
- 来源弱但方向有价值的线索；
- 暂时无法确认主体的信息；
- 与预测相关但重要性不足的资料；
- 暂时没有模型节点接收的信息；
- 查询未得记录；
- 低置信但可能成为课题种子的观察。

## 10.2 触发机制

冷存不是垃圾箱。它必须有触发机制：

- 同主体同方向累计超过阈值；
- 与当前高敏感假设相关；
- 新硬锚出现后可验证；
- 被课题查询命中；
- 某个 driver 转 contested；
- 临近财报；
- 人工手动激活。

触发后，冷存信息可以：

- 重入 L2；
- 进入证据账本；
- 生成课题候选；
- 生成待办；
- 继续冷存。

## 10.3 Tier 3 的处理

自媒体、大 V、论坛、社群传闻原则上先进冷存。

它们可以作为：

- 早期信号；
- 检索线索；
- 课题触发器；
- 与其他证据的辅助印证。

但不能单独支撑正式预测里的 accepted claim。

# 11. 人机待办

## 11.1 定位

人机待办不是普通 todo list，而是系统把“不确定性”和“研究缺口”转化为人类行动的界面。

待办必须落到变量级，不能只写“研究一下某公司”。

## 11.2 待办排序

排序公式可以概念化为：

```text
预测影响 × 置信度缺口 × 财报临近度 × 可解决性 × 分歧价值
```

其中：

- 预测影响：解决后可能改变多少净利润预测；
- 置信度缺口：当前证据距离正式采信差多少；
- 财报临近度：越接近验证点越优先；
- 可解决性：是否能通过公告、API、专家、卖方快速解决；
- 分歧价值：不同情景下净利润差异多大。

## 11.3 人类负责什么

人类主要负责：

- 批准新增个股初始化；
- 批准高成本课题；
- 裁决高重要性冲突；
- 批准口径契约变化；
- 批准核心框架变化；
- 做物理世界调研；
- 问卖方、专家、公司；
- 判断是否发布正式快照；
- 覆盖机器假设并给出理由。

## 11.4 机器负责什么

机器主要负责：

- 整理资料；
- 抽取 claims；
- 做初步校验；
- 发现冲突；
- 对账；
- 建议待办；
- 生成课题信息需求清单；
- 查账本和冷存；
- 主动检索；
- 生成影子预测；
- 生成复盘归因草稿；
- 提出框架 / 规则 diff。

## 11.5 人工裁决记录

每次人工裁决都要记录：

- 裁决对象；
- 裁决结果；
- 理由；
- 使用资料；
- 是否覆盖机器判断；
- 影响哪些 claims；
- 影响哪些假设；
- 是否触发重算；
- 是否应回写规则或框架。

人工判断不是系统外事件。它本身也是系统学习资料。

# 12. 执行日志、版本和快照

## 12.1 执行日志 M-LOG

执行日志记录系统做过什么。

至少包括：

- 何时执行；
- 由什么触发；
- 执行了什么行动；
- 输入是什么；
- 输出是什么；
- 加载了哪些规则工件；
- 引用了哪些框架条目；
- 使用了哪些 source_ref；
- 产生了哪些 claims；
- 改了哪些假设；
- 是否进入失效队列；
- 是否生成快照；
- 是否生成待办；
- 是否失败。

执行日志是后续补血缘、做复盘、训练框架的基础。

## 12.2 版本引用

正式快照和关键中间结果至少要记录：

- 证据版本；
- claim 采信视图版本；
- 口径契约版本；
- 框架条目版本；
- 规则工件版本；
- 模型版本；
- 假设版本；
- 数据源追溯等级；
- 执行日志 ID；
- 发布人 / 发布动作。

## 12.3 不可变与可变

不可变：

- 自维护 raw 原文；
- source_event；
- 原始 claims；
- 执行日志；
- 发布后的预测快照；
- 财报复盘原始记录。

可变视图：

- 当前采信 claims；
- 当前假设；
- 当前预测模型实例；
- 当前待办状态；
- 当前框架最新版；
- 当前规则最新版。

“可变视图”变化时必须能追到历史版本。

## 12.4 快照不可回改

如果发现旧预测错了，不改旧快照。

正确做法是：

- 生成新快照；
- 或生成重述复盘；
- 或标记旧快照受某个后验信息影响；
- 但不覆盖原始发布记录。

# 13. 冲突、置信度和情景管理

## 13.1 冲突类型

系统至少识别：

- 数值冲突；
- 时间冲突；
- 口径冲突；
- 主体冲突；
- 来源冲突；
- 事实与观点混用；
- 估算和披露锚冲突；
- 预测与实际披露冲突；
- 框架解释冲突；
- 人工判断与机器判断冲突。

## 13.2 置信度来源

置信度不能只由 LLM 打分。

它应该综合：

- 来源 Tier；
- 追溯等级；
- 与硬披露锚的一致性；
- 是否多源印证；
- 来源历史准确率；
- 信息是否直接；
- 是否转述；
- 是否有明确期间和口径；
- 抽取校验是否通过；
- 是否被财报验证；
- 是否有人裁决。

## 13.3 冲突处理

冲突处理不是简单选一个。

处理方式包括：

- 降置信；
- 转 contested；
- 生成待办；
- 生成情景分支；
- 暂时采信硬锚；
- 等待新资料；
- 请求人工裁决；
- 阻塞正式快照；
- 只允许影子预测。

## 13.4 情景分支

情景分支只服务于预测区间和不确定性表达，不是让系统逃避判断。

系统仍然需要有当前主假设。情景分支用于表达：

- 关键变量存在分歧；
- 不同分歧对净利润影响很大；
- 暂时没有足够证据裁决；
- 需要把不确定性显性化。

# 14. 财报季复盘与系统学习

## 14.1 财报是系统的验证集

财报季复盘不是写点评，而是系统训练和纠错过程。

财报披露后，系统要回答：

- 预测错在哪里；
- 是哪个节点先偏离；
- 是证据错、抽取错、口径错、框架错，还是人工判断错；
- 哪些规则应该更新；
- 哪些来源应该降权；
- 哪些公司模型应该修改；
- 哪些待办应该长期保留。

## 14.2 逐节点复盘

不能只看净利润误差。

必须沿利润桥复盘：

```text
收入
→ 毛利率
→ 毛利
→ 费用
→ 经营利润
→ 财务费用 / 投资收益 / 其他收益
→ 税率
→ 少数股东损益
→ 归母净利润
```

现金质量也要同步看：

- 利润是否转化为现金；
- 应收是否异常；
- 存货是否异常；
- 合同负债是否验证需求；
- 资本开支是否验证扩产；
- 是否出现减值或一次性项目。

## 14.3 误差归因分类

建议初始分类：

1. 来源错误；
2. 抽取错误；
3. 口径错误；
4. 对账失败；
5. 仲裁错误；
6. 假设方向错误；
7. 假设节奏错误；
8. 模型结构错误；
9. 框架条目错误；
10. 规则工件错误；
11. 人工覆盖错误；
12. 突发事件；
13. 公司披露变化；
14. 蒙对但逻辑错误；
15. 蒙错但方向合理。

## 14.4 回写

复盘结果要回写到：

- 来源可信度；
- 抽取规则；
- 对账规则；
- 口径契约；
- 框架条目；
- 公司驱动树；
- 假设账本；
- 课题模板；
- 人机待办排序；
- 未来快照发布规则。

复盘如果不回写，系统不会变好。

# 15. 当前阶段的工程取舍

这不是路线图，而是当前架构必须承认的现实边界。

## 15.1 L1 先轻后重

当前不需要先做全量 raw data lake。

优先做：

1. 公告 / 财报 PDF 自维护；
2. 资料来源登记；
3. 查询和访问记录；
4. 追溯等级；
5. 与 L2/L3 的证据链。

其他资料通过 API、MCP、web、人工上传进入，不强求原文级保存。

## 15.2 先把证据账本和预测快照做硬

比起大而全的数据层，早期更重要的是：

- claim 能入账；
- claim 能追到来源记录；
- claim 能冲突；
- claim 能影响假设；
- 假设能影响预测；
- 预测能生成快照；
- 快照能复盘。

## 15.3 baseline claims 暂不启用

一致预期、卖方预测、市场 consensus 可以以后作为对照模块。

当前阶段不把它放入核心流程，避免系统被一致预期牵着走。

## 15.4 不依赖实时在线

系统按事件语义设计，但当前运行形态可以是：

- 定时批处理；
- 人工拉起；
- session 式 agent harness；
- 局部重算；
- 影子快照常算；
- 正式快照人工过闸。

不要求第一天实时流式运行。

## 15.5 harness 无关

可以用 Claude、Codex、MCP、Python、数据库、搜索工具等执行行动，但系统控制逻辑不能绑定某个 harness。

控制逻辑应该沉淀在：

- 框架条目；
- 规则工件；
- 口径契约；
- 行动日志；
- 快照约定；
- 待办机制。

# 16. 硬边界

以下边界建议锁定。

1. 系统唯一正式产出是公司未来财务预测；当前阶段是契约口径下净利润预测。
2. 系统不做估值、选股、买卖建议、便宜判断、组合构建。
3. L1 是资料接入与披露锚归档层，不是必须自有全量 raw data 仓库。
4. 上市公司公告 / 财报 PDF 是自维护硬锚，原文尽量保存。
5. API、MCP、web、购买数据服务可以只保留来源标识、查询参数、访问时间和返回摘要。
6. 所有进入预测的证据必须有追溯等级；弱追溯必须有置信度上限。
7. 事实、观点、估算、预测、传闻、关系、口径、框架、规则、假设、输出必须分开。
8. 证据账本是核心长期资产；derived claims 属于证据账本，不得形成第二套真相。
9. 披露为锚，分硬披露锚和软披露锚；外部数据必须与锚解释得通。
10. 对账规则放在规则工件架，L2/L6 只是加载执行。
11. 口径契约第一天存在，可以很轻；采用全局基准 + 公司附录。
12. 历史重构、未来预测、财报复盘必须引用同一版口径契约。
13. 知识与规则层是旁路；框架架和规则工件架分开，但都条目化、版本化、可引用。
14. 玩家、客户、供应商、竞争关系属于关系型 claims，不直接放进框架层。
15. 预测快照必须记录框架条目和规则工件引用，不能只记一个总版本号。
16. 账本、日志和快照必须支持 as-of 复盘。
17. 净利润阶段也必须保留完整利润桥，不允许只有一个净利润数字。
18. 现金质量检查必须存在，即使早期只影响置信度和待办。
19. 冲突不能吞掉；高重要性冲突阻塞正式快照，但不阻塞影子快照。
20. 情景分支只给高重要性 contested driver，且数量受限。
21. 能确定性完成的任务不交给 LLM；LLM 评分只能作为准入特征之一。
22. 生产和校验隔离；校验 agent 只看原文和产出，不看生产推理。
23. 调度脊柱采用感知 / 触发 / 准入 / 行动 / 队列结构。
24. 血缘第一阶段按最小集建设，其余依赖由执行日志保留。
25. 上游变化通过订阅和血缘局部重算，不做无脑全量重算。
26. 课题是复合行动，不是新层；必须绑定目标变量。
27. 课题主动获取的资料必须进入 L2 和 L3，不能绕过账本直接改预测。
28. 冷存 / 弱信号池必须有触发机制，否则只是垃圾箱。
29. 人工裁决必须记录理由和影响路径，并回流为规则和框架优化输入。
30. 财报季复盘是流程，不是报告；必须逐节点归因并回写上游。
31. 一致预期 baseline 暂时作为可选槽位，不进入第一阶段核心。
32. demo 期允许定时 / 人工拉起的批式运行，不要求实时在线。
33. 系统控制逻辑必须 harness 无关。
34. 快照发布后不可回改，只能生成新快照或重述复盘。

# 17. 术语表

| 术语 | 含义 |
|---|---|
| 资料接入层 | 负责资料访问、来源登记、披露锚归档、检索记录的层 |
| 披露锚 | 财报、公告、问询函等高优先级披露资料 |
| claim | 关于世界的一条可判断陈述，是证据账本的基本单位 |
| derived claim | 由原始 claims 通过口径契约或计算规则派生的 claim |
| 当前采信 claim | 系统当前认为可进入假设或预测的 claim 视图 |
| 追溯等级 | 某条证据能追溯到原文、数据服务、网页、人工记录或弱来源的程度 |
| 口径契约 | 规定历史重构、未来预测、财报复盘按什么口径处理的规则工件 |
| 框架条目 | 行业、公司、技术、商业模式、财务预测等解释结构的版本化条目 |
| 规则工件 | 清洗、抽取、对账、仲裁、准入、课题、复盘等操作规则 |
| 驱动树 | 将公司业务和财务科目连接到预测节点的结构 |
| 假设账本 | 记录模型当前采用的输入假设及其来源、置信度、冲突状态 |
| 影子快照 | 系统自动生成、用于观察和比较的非正式预测快照 |
| 正式快照 | 通过发布闸门、进入正式输出的预测快照 |
| 调度脊柱 | 感知、触发、准入、行动、队列的运行控制结构 |
| 课题 | 绑定目标变量、主动寻找证据和更新模型的复合行动 |
| 冷存池 | 暂时不足以采信但可能累计触发的弱信号集合 |
| M-LOG | 执行日志，记录系统每次行动的输入、输出、版本和结果 |
| as-of 复盘 | 回到某一历史时点，只用当时可得信息评价当时预测 |

# 18. 一句话收口

这个系统不是“更强的搜索助手”，而是一个围绕财务预测长期运转的证据、规则、假设和复盘机器。

L1 可以现实地轻一些：公告 PDF 自维护，其他来源保留访问与追溯记录。真正不能轻的是 L3 证据账本、L5 预测快照、L6 调度脊柱和财报复盘。只要进入预测，信息就必须能被解释、被追溯、被冲突、被重算、被复盘。
