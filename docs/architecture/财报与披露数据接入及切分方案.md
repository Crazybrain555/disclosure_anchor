# 财报与披露数据接入及切分方案 v1.0

> 本文件定义财务数据与披露文件如何进入投研预测引擎，以及财报、投关记录和公告应如何被切成 L2 可直接使用的对象。它是 `disclosure_anchor` 的**接入与切分实施方案**：术语、边界和硬决策以 `service-purpose.md`（canonical 契约）为准，本文件在其框架内给出标准数据 provider 分工、PDF 补充范围、切分细则和实施顺序。数据库、解析和接口实施文档应同时遵循两者。

# 0. 执行摘要

第一版采用两条互补的数据路径：

```text
标准、可复用、已有 provider 覆盖的数据
→ Dataset API
→ 直接交给 L2 / L3

自维护公告和财报 PDF
→ disclosure_anchor
→ document + document_unit
→ 交给 L2
```

核心决策如下：

1. 标准数据 provider（Wind / Tushare / 同花顺 iFinD / Choice 等）和 PDF 属于同一逻辑 `L1`，但不塞进同一种物理表；具体用哪家 provider 可替换，不影响架构。
2. provider 已覆盖的三大表、标准财务指标、审计意见等，不从 PDF 重建。
3. PDF 主要补充经营细分、财务附注、管理层解释、问答和重大事项原文。
4. PDF 文本需要切分，但按章节、子标题、完整问答和完整表格切，不按页码或 token 切。
5. 切分结果统一称为 `document_unit`，只有 `text / table / qa` 三种。
6. `heading_path` 是逻辑标题路径，不是 Markdown 文件地址；回到 parser artifact 使用可选 `artifact_locator`。
7. 表格第一阶段保留完整结构，不拆 cell，不急着建立全市场标准表。
8. 只有经过实际复用证明的表族，才从 `table unit` 晋级为标准 dataset。
9. 短公告不在 L1 生成 canonical `event_unit`；事件由 L2 抽取。
10. 第一版删除 page、bbox、持久化 chunk、table_cell、多 parser 常态对账和跨 parser 锚迁移。
11. parser 或切分形态变化不触发 L3；真实内容、数字或来源值变化才触发。
12. 原始 PDF 和 parser artifact 保留，派生层可以大胆去噪和精简。

这套方案的目标不是完整复制 PDF，而是：

> 以尽可能小的数据库和解析复杂度，让 L2 稳定找到 L3/L5 当前真正需要的数据与原文。

# 1. 总体架构

## 1.1 L1 的两条主路径与其他来源

| L1 标准数据路径 | L1 披露文件路径 | L1 其他来源（非标，旁路） |
| --- | --- | --- |
| Wind / Tushare / 同花顺 iFinD / Choice / API | CNINFO / 交易所 PDF | 非标 API / MCP / Web / 搜索 |
| Dataset Registry → Provider Adapter → `Dataset API` | `disclosure_anchor` → `document` / `document_unit` → `Filing API` | 通用 `source_access`（直接进 L2，不建 `document_unit`） |

三条路径最终都在 `L2` 汇合 →（抽取 claim / 事件 / 口径 / 冲突）→ `L3` 证据账本。

前两条是主路径，二者在职责上同层：都是来源接入与可用化。

二者在数据形态上不同：

- 标准数据 provider 返回标准记录；
- PDF 返回文档结构单元。

因此统一发生在访问契约和 L2 消费层，而不是底层数据库表。

（第三列“其他来源”是更轻的旁路：非标 API、MCP、Web、搜索经通用 `source_access` 直接进 L2，不强求转成 `document_unit`，详见 §1.2。）

## 1.2 第三类来源

Web、MCP、搜索、新闻和其他一次性 API 返回，可以通过通用 `source_access/source_ref` 进入 L2。

第一版不要求把这些来源全部转成 `document_unit`，也不要求 `disclosure_anchor` 承担它们。

# 2. 标准数据 provider 与 PDF 的分工

> **标准数据侧是 provider 无关的。** 下文以 Wind 为具体示例，但 Wind 只是首版采用的一个 provider，可整体或按 dataset 替换 / 并存为 Tushare、同花顺 iFinD、Choice 或其他数据源。抽象由 `dataset_registry`（语义 `dataset_key`）+ `provider_adapter`（各家参数与字段映射）承担：`dataset_key` 与具体 provider 解耦，下游 L2 只认 `dataset_key` 和返回模型，不感知背后是哪家。§2.1 的 `AShare*` 表名是 Wind 的内部对象，换 provider 时由对应 adapter 映射到相同的 `dataset_key`。

## 2.1 标准数据 provider 作为 operational primary（以 Wind 为例）

以首版采用的 Wind 为例，标准财务数据覆盖如下。换用其他 provider 时，由各自 `provider_adapter` 映射到相同的 `dataset_key`，下游不感知差异。

## 2.2 第一版不从 PDF 重建这些数据

以下内容默认不建立第二套本地标准表：

- 三大财务报表；
- 标准财务指标；
- TTM / MRQ 和衍生指标；
- 业绩预告、业绩快报的标准字段；
- 审计意见；
- 会计变更；
- 银行、券商、保险等专用标准指标。

PDF 中仍保留原文和 parser artifact，但默认不把这些表转换成核心 `document_unit(table)`。

## 2.3 不做日常双源核验

第一版把配置的标准数据 provider（首版为 Wind）当作日常使用的默认来源：

```text
operational_primary = <标准数据 provider>   # 首版配置为 Wind
```

不要求每次都重新解析 PDF，也不要求逐项双源对账。

只有出现以下情况才回到正式披露：

- provider 缺字段；
- 更正公告或追溯重述；
- 合并 / 母公司口径疑问；
- 报告期或单位异常；
- 关键数字明显不合理；
- L2/L3 需要公司原文解释；
- 重要预测节点需要加强复核。

因此正式披露是：

```text
authoritative_fallback
```

而不是常态重复数据源。

## 2.4 provider 数据的 as-of 快照

不需要把 provider 全库复制到本地，但进入 claim、假设或预测的值必须保存当时快照：

```text
dataset_key
provider
query params
queried_at
field
value
unit
period
scope
source_access_id
```

这样以后 provider 修订历史值时，仍能复盘当时预测使用了什么。

# 3. PDF 应该补什么

## 3.1 经营细分表

优先提取：

- 分行业收入、成本和毛利率；
- 分产品收入、成本和毛利率；
- 分地区、分渠道和分客户类型；
- 产量、销量和库存量；
- 成本构成；
- 前五大客户和供应商集中度；
- 主要子公司和分部经营数据；
- 研发项目、研发投入和人员；
- 产能、扩产项目和资本开支。
等其他重要内容（以上只是我初步设想的）


这些表通常结构可复用，但公司维度值不同，因此第一阶段保存为完整 `table unit`，不建立统一产品字典。

## 3.2 财务附注表

优先提取：

- 应收账款账龄；
- 坏账准备和计提比例；
- 存货分类和跌价准备；
- 固定资产和在建工程；
- 债务结构及到期情况；
- 资产减值；
- 外币资产负债敞口；
- 重要子公司、联营企业和分部；
- 合并范围变化；
- 重大承诺、或有事项和关联交易；
- 与当前驱动树直接相关的其他附注。
等其他重要内容（以上只是我初步设想的）

这些表先作为 `table unit`，之后根据使用频率晋级。

## 3.3 管理层解释和业务文本

优先提取：

- 主营业务和产品结构；
- 行业变化；
- 价格、销量、订单、客户和渠道变化；
- 产能利用率、扩产和投产节奏；
- 毛利率、费用、现金流和资产负债变化原因；
- 技术、认证和新产品进展；
- 并购、合并范围和组织变化；
- 关键审计事项；
- 风险；
- 未来展望；
- 与个股预测驱动树相关的其他解释。
等其他重要内容（以上只是我初步设想的）

这些内容保存为 `text unit`。

## 3.4 投资者关系问答

投关活动记录、业绩说明会和公开交流按完整问答保存为 `qa unit`。

```text
一个 Question + 完整 Answer = 一个 qa unit
```

L1 不因为答案很长而按 token 拆碎。

## 3.5 短公告和重大事项

分红、回购、定增、股权激励、重大合同、产能投资、并购等公告，第一版仍切成 `text/table` 单元。

事件结构化由 L2 完成：

```text
text/table unit
→ L2 识别 event type 和字段
→ event claim / 事件对象
```

这样 L1 不会逐渐膨胀成事实层。

# 4. 标准化的三段式路径

用户提出的原则“可复用就标准化，不可复用就按主题保存”是正确方向，但第一版需要增加中间态。

## 4.1 A 类：provider 已覆盖的标准数据

直接通过 Dataset API 使用。

例子：

```text
financial.income_statement
financial.balance_sheet
financial.cash_flow
financial.indicators
financial.audit_opinion
```

不进入 `document_unit`。

## 4.2 B 类：可复用但尚未证明值得标准化的表族

先保存为 `table unit`。

例子：

```text
产品收入和毛利率
产销量和库存量
应收账款账龄
存货分类
在建工程明细
前五大客户
```

它们具有稳定的表语义，但公司业务维度、表头层级和披露细节不完全统一。

第一阶段只要求：

- 表能完整读出；
- 标题和单位清楚；
- 行列关系不丢；
- L2 能检索和引用。

## 4.3 C 类：公司特定或难复用的内容

保存为 `text` 或 `qa`。

例子：

- 某产品认证节奏；
- 特定客户导入；
- 某产线的技术改造；
- 管理层对库存增加的解释；
- 公司特有技术路线。

## 4.4 table unit 晋级标准 dataset 的条件

只有同时满足以下条件才晋级：

1. 多家公司或多期反复出现；
2. 字段语义相对稳定；
3. L2/L3/L5 反复查询；
4. 标准数据 provider 没有可靠覆盖；
5. 抽取质量可通过简单规则校验；
6. 标准化后的收益大于 schema 和维护成本。

晋级后：

```text
table unit 仍然保留
+ 新增一个 dataset adapter / standardized view
```

标准 dataset 不覆盖原始文档单元。

首批可能晋级的候选：

- `financial_notes.receivable_aging`；
- `financial_notes.inventory_composition`；
- `operating.volume_sales_inventory`；
- `operating.segment_revenue`；
- `operating.capex_projects`。

是否晋级以真实调用记录为准，不在第一版预判。

# 5. document_unit 模型

## 5.1 定义

`document_unit` 是披露文件中最小的 L1 可寻址结构单元。

它不是：

- page；
- parser block；
- RAG chunk；
- table cell；
- claim；
- event fact。

一条 unit 可以在 L2 中产生零条、一条或多条 claims。

## 5.2 通用契约

```text
document_unit_id
 document_id
processing_run_id
unit_kind                 text / table / qa
heading_path              逻辑标题路径
title
order_index
semantic_key              可选的粗粒度语义键
payload                   正文 / 表格 / 问答
content_hash
quality_status
artifact_locator          可选
```

## 5.3 heading_path

`heading_path` 是标题面包屑，而不是 Markdown 文件地址。

例如：

```json
[
  "第三节 管理层讨论与分析",
  "一、报告期内公司从事的主要业务"
]
```

它是 agent 和 L2 的逻辑地址。

## 5.4 artifact_locator

`artifact_locator` 是可选的技术地址，可用于回到：

- Markdown；
- parser JSON；
- 表格 HTML；
- 其他解析产物。

它可以只保存：

```text
artifact_path + artifact_unit_ref + order_index
```

第一版不要求 page、bbox 或字符偏移。

## 5.5 text payload

```json
{
  "text": "机械清洗后的完整正文"
}
```

## 5.6 table payload

以江海股份（002484）2025 年年度报告“应收账款 / 按账龄披露”为例：

```json
{
  "unit": "元",
  "headers": ["账龄", "期末账面余额", "期初账面余额"],
  "rows": [
    ["1 年以内（含1 年）", "1,765,831,017.43", "1,653,778,854.38"],
    ["1 至2 年", "23,872,757.96", "35,360,192.57"],
    ["2 至3 年", "14,382,374.82", "7,076,511.89"],
    ["3 年以上", "73,573,362.32", "78,454,116.92"],
    ["3 至4 年", "1,443,796.67", "11,272,000.94"],
    ["4 至5 年", "5,272,677.17", "450,708.56"],
    ["5 年以上", "66,856,888.48", "66,731,407.42"],
    ["合计", "1,877,659,512.53", "1,774,669,675.76"]
  ],
  "notes": [],
  "nearby_explanation": ""
}
```

表格保留原始字符串。第一版不要求把每个值转换成统一 metric。

> 注意原表中“3 年以上”为小计行，其下再拆“3 至4 年 / 4 至5 年 / 5 年以上”三个子区间；下游做“合计 = 各账龄之和”校验时应避免重复计数。这类小计与明细并存的结构由 L2 在使用时识别，L1 只忠实保留原始行。

## 5.7 qa payload

```json
{
  "question": "美国加征关税对公司有什么影响？",
  "answer": "完整回答"
}
```

会议时间、地点、参与人和活动类型放在 `document` 元数据中，不必在每个问答中重复。

# 6. 文本和表格的切分规则

## 6.1 结构边界优先级

```text
文档标题
→ 章节
→ 子章节
→ 显式编号条目
→ 完整 Q&A
→ 完整表格
→ 完整事项说明
```

## 6.2 不采用的边界

第一版不按以下方式持久化切分：

- page；
- 固定字符数；
- 固定 token 数；
- overlap；
- parser block；
- 每个单元格。

## 6.3 长文本

存在真实子标题时继续按子标题切。

没有更细真实结构时，即使内容较长，也保持一个完整 `text unit`。agent 运行时可以临时截取，不创建长期 chunk。

## 6.4 跨页表

跨页只是 PDF 版面问题。解析后应合并为一张逻辑表，生成一个 `table unit`。

调用方不需要知道它来自几页。

## 6.5 相邻解释

表格标题、单位、脚注和紧邻的解释应尽量跟表放在同一个 `table unit`。

独立的管理层解释另建 `text unit`。

# 7. 不同文件类型的处理方案

## 7.1 年报、半年报和季报

处理流程：

```text
识别标题树
→ 应用保留策略
→ 保留高价值 text sections
→ 提取目标 table families
→ 跳过标准数据 provider 已覆盖的标准表
→ 生成 text/table units
```

正文保留范围以当前净利润预测驱动树为核心，不追求所有章节全量入库。

## 7.2 投资者关系活动记录和业绩说明会

```text
文档元数据
+ Q1 / A1
+ Q2 / A2
+ ...
```

每个完整问答一个 `qa unit`。

一个答案中的多个数字、观点和时间范围由 L2 分拆。

## 7.3 业绩预告和业绩快报

- 标准区间、净利润等字段优先从标准数据 provider 使用；
- PDF 重点保留变动原因、一次性项目、口径说明和管理层解释；
- provider 缺失或需要原文时，再由 L2 从 unit 中抽取字段。

## 7.4 分红、回购、定增和股权激励公告

第一版按显式章节生成少量 `text/table units`。

不在 L1 直接形成 canonical event。L2 可以从中抽取：

- 分配基数；
- 每股金额；
- 登记日和除权日；
- 回购数量和价格；
- 募集规模；
- 激励对象和考核条件等。

## 7.5 问询函和回复

优先按“问题 + 完整回复”形成 `qa unit`；回复中的独立表格形成 `table unit`。

## 7.6 普通重大事项公告

短公告通常只包含一个事项：

- 一个主 `text unit`；
- 必要时附一个或多个 `table unit`。

公告中存在多个真正独立事项时，再按显式标题拆分。

# 8. 财报的保留和去噪策略

## 8.1 默认保留

### 管理层讨论与分析

- 主营业务；
- 行业情况；
- 产品与业务变化；
- 经营数据解释；
- 收入、成本、毛利率和费用分析；
- 现金流和资产负债变化；
- 重大投资和资本开支；
- 风险和未来展望。

### 高价值表格

- 分产品 / 行业 / 地区收入和毛利率；
- 产销量和库存量；
- 成本构成；
- 客户和供应商集中度；
- 应收账款、存货、固定资产、在建工程和债务；
- 分部和子公司数据；
- 与当前驱动树相关的其他表。

### 重大事项

- 合并范围变化；
- 并购重组；
- 资产减值；
- 重大合同；
- 扩产；
- 诉讼和或有事项；
- 会计差错和更正。

## 8.2 默认不生成 document_unit

- 封面；
- 目录；
- 页眉、页脚和页码；
- 固定责任声明；
- 联系方式和备查文件；
- 签章；
- 空表，以及只有表头、单位而无实质数据的表；
- 只有“适用 / 不适用”“是 / 否”勾选而无实质内容的模板；
- 财务报表附注中照抄会计准则的套话段落（重要会计政策的一般性表述、各会计科目确认计量原则、金融工具 / 收入 / 租赁 / 合并报表编制等标准定义，以及“遵循企业会计准则的声明”“重要性判断标准”等模板）——但会计政策 / 会计估计的**实际变更及其影响**仍按 8.1 保留；
- 独立董事、监事会、保荐机构、会计师等标准格式意见中无个性化结论的模板段落；
- 重复公司宣传，以及重复出现的免责、版权和前瞻性陈述提示；
- 当前预测系统不使用的纯形式性披露；
- 标准数据 provider 已覆盖的标准报表和指标。

这里的“不生成”不等于删除。原 PDF 和 parser artifact 仍然保留，规则升级后可重跑。

> 8.1 / 8.2 都是**示意性清单，不是穷尽的硬编码规则**，说明的是“按实质内容判断”的取舍倾向，而非要求逐条照搬。两个原则优先于具体条目：一是**有实质信息就保留、纯格式和重复套话才跳过**；二是**拿不准时倾向保留**——漏掉实质披露的代价远高于多生成一个单元。同一类标题在不同文档里可能落在不同侧（例如“风险提示”，个性化、可量化的风险应保留，纯模板套话才跳过），应结合 §8.3 的规则边界按文档类型和语义判断，不要机械匹配字面标题。

## 8.3 保留策略的实现

第一版使用：

```text
filing_type rules
+ heading allowlist / denylist
+ table semantic rules
+ deterministic cleaning
+ optional coarse classifier
```

不得在 L1 让 LLM 自由决定投资重要性。

不确定内容可以：

- 暂不生成 unit；
- 保留在完整 parser artifact；
- 后续规则升级后重跑。

# 10. 数据源目录和适配器

## 10.1 dataset_registry

统一数据源能力的 canonical 定义应放在 Git 中，而不是首先做成一张人工维护的数据库表。

每个 dataset contract 描述：

```text
dataset_key
description
query model
result model
primary provider
fallback provider
refresh policy
schema version
owner
status
```

示例：

```yaml
key: financial.income_statement
description: A股利润表
primary_provider: wind          # 示例；可换 tushare / ifind / choice，dataset_key 与 provider 无关
provider_object: AShareIncome
fallback_provider: official_filing
query_fields: [company, period, scope]
result_model: IncomeStatement
```

```yaml
key: disclosure.receivable_aging
description: 应收账款账龄和坏账计提
primary_provider: official_filing
provider_object: document_unit
query_fields: [company, period]
result_model: TableUnit
```

## 10.2 provider_adapter

每个 provider 负责：

```text
统一查询参数
→ provider 参数转换
→ 获取数据
→ 转换为统一返回模型
→ 记录 source_access
```

第一版可以只是普通 Python adapter，不必立即建设复杂框架。

## 10.3 source_access

数据库只记录实际发生的访问：

- dataset key；
- provider；
- 查询参数；
- 访问时间；
- 状态；
- 结果哈希；
- 错误；
- 必要的返回快照或定位。

接口文档、字段定义和 provider 映射由 Git 中的 registry 管理。

# 11. 最小数据库

## 11.1 第一版核心对象

```text
company
security
source_access
document
processing_run
document_unit
optional source_checkpoint
```

## 11.2 company / security

维护公司和证券基础关系。公司主体与证券代码分开。

## 11.3 source_access

覆盖标准数据 provider（Wind / Tushare / 同花顺 iFinD / Choice 等）、CNINFO、MCP、Web 和其他 API 的实际调用记录。

## 11.4 document

用于自维护披露文件：

- provider document id；
- 公司和证券；
- 公告标题和类型；
- 公告日期和报告期；
- source_access；
- 原文件路径和哈希；
- 当前状态；
- 更正和替代关系。

## 11.5 processing_run

记录下载、解析、清洗和结构切分的运行版本和状态。

## 11.6 document_unit

只服务披露文档，保存 `text/table/qa`。

标准数据 provider 的记录不写入 `document_unit`。

## 11.7 source_checkpoint

仅在增量同步需要游标时建立。

## 11.8 文件系统

```text
raw_documents/
  原始 PDF

parser_artifacts/
  Markdown
  parser JSON
  table HTML / JSON
  其他调试产物
```

> `document_unit` 存的是**切好的内容快照本身**（在 `payload` 里），不是只存地址：数据库保存原文件路径、哈希、检索索引，以及每个单元的 `payload` 快照。`artifact_locator` 只是可选的回溯指针，用于回看原文核对；查询和证据引用都依赖 `payload` 快照。这也是为什么内容固化后不需要 parser block 镜像表 / table_cell / persisted chunk 等回指结构（见 §11.9）。

## 11.9 第一版不建设

- `filing_text_block`；
- `filing_text_chunk`；
- `filing_table_cell`；
- page / bbox locator；
- cross-parser alignment；
- 独立向量库；
- 图数据库；
- 全量标准数据 provider 镜像仓库；
- 每种财务附注表一张 SQL 表。

# 13. 追溯、版本和 as-of

## 13.1 披露来源

一个正式披露 claim 可以引用：

```text
source_access_id
+ document_id
+ raw_file_hash
+ processing_run_id
+ document_unit_id
+ exact snapshot
```

无需强制 page 和 bbox。

## 13.2 标准数据来源

标准数据 provider claim 引用：

```text
dataset_key
+ provider
+ query params
+ queried_at
+ returned field/value snapshot
```

## 13.3 G0 建议定义

披露侧 G0 建议定义为：

```text
自维护原文件
+ 文件哈希
+ 精确 document_unit / 表格快照
```

页码或视觉区域是附加复核信息，不是必要条件。

## 13.4 版本策略

- 原始文件版本不可变；
- parser run 可重跑；
- unit 属于具体 run；
- 当前查询默认返回 active run；
- 旧 claim 保留旧 run 和 exact snapshot；
- 不要求新旧 parser unit ID 一致。

# 14. 变更传播

## 14.1 不触发 L3

- page 变化；
- bbox 变化；
- Markdown 样式变化；
- parser 版本变化；
- 逻辑 unit 边界变化；
- 运行时上下文截取变化；
- 同一内容的格式差异。

## 14.2 触发 L2 重新处理

- 新公告；
- 更正公告；
- 原文件哈希变化；
- unit 正文变化；
- 表头、行、单位或数字变化；
- 新保留规则识别出新的候选单元；
- 解析质量状态改善；
- 标准数据 provider 历史值修订。

## 14.3 触发 L3/L5 的条件

只有当重新处理形成：

- 新 claim；
- claim 值变化；
- claim 状态变化；
- 假设变化；
- 口径契约变化；

才沿血缘触发预测更新。

parser 几何变化不直接触发预测。

# 15. 解析和质量策略

## 15.1 默认 parser

第一版使用一个默认解析后端即可。当前调研可继续采用 MinerU 作为默认后端。

Docling、Camelot 或其他工具只在以下情况按需启用：

- 默认 parser 失败；
- 关键表结构明显错误；
- 某张表即将进入正式预测，需要增强复核；
- 对特定文档类型形成专用 fallback。

## 15.2 轻量质量检查

正文：

- 文本非空；
- 标题顺序合理；
- 断行和乱码可接受；
- Q&A 边界完整。

表格：

- 有标题或可识别语义；
- 有表头和数据行；
- 单位可识别或标记缺失；
- 行列数合理；
- 数字解析率合理；
- 有显式合计时做简单加总检查。

不通过时标记：

```text
needs_review
unusable
```

## 15.3 不做的质量工程

- 全量双 parser；
- 所有表逐 cell 对账；
- 所有 PDF 数字和标准数据 provider 逐项核验；
- 所有错误自动修复；
- 所有视觉坐标正规化。

更严格的复核由“是否真的进入重要 claim”决定。

# 16. 实施顺序

## Phase 0：锁定当前预测需求

1. 从净利润驱动树列出当前真正需要的数据；

1. 使用 CNINFO 已验证接口维护 company/security；
2. 同步公告分类和公告索引；
3. 下载重要 PDF；
4. 保存 document 和 raw hash；
5. 建立 processing_run。

第一阶段可继续使用：

- `p_stock2100`；
- `p_stock2101`；
- `p_info3005`；

1. 年报：江海股份；


```text

```

## Phase 5：运行后晋级

运行一段时间后，根据实际调用频率：

- 晋级高复用 table family；
- 增加少量 semantic key；
- 增加专用抽取器；
- 只对明确瓶颈增加全文或语义检索。

# 17. 第一版验收标准

1. 三大表和标准财务指标从标准数据 provider 获取，不从 PDF 重建第二套数据；
2. 江海股份年报的主营业务和经营分析可按 heading path 取得；

5. 权益分派公告可被 L2 抽取成分红事件；

9. parser 版本变化但内容不变时，不触发 L3；

12. 下载、解析和切分失败可定位、可重试；

14. 不上向量数据库也能完成主要检索和 L2 消费闭环。

# 20. 最终架构结论

最合适的第一版不是“把所有 PDF 变成一个完整数据库”，也不是“把所有内容都做成 RAG chunk”。

正确形态是：

