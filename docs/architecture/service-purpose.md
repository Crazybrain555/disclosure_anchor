# disclosure_anchor 服务目的

参考协议：`docs/reference/投研预测引擎顶层框架协议_v0.5.md`

## 一句话

`disclosure_anchor` 是一个本地上市公司公告 / 财报数据库服务，定位是投研预测引擎里的 **L1 披露锚归档层**：保存可复查的原文与解析结果，并把它切成“可寻址、可发现重点”的披露锚，让 L2 一抓就到、一看就知道重点在哪。它本身不产出事实、不下判断、不采信。

它不是只把 PDF 下载到文件夹里，也不是一个 RAG 问答库，也不是要替 L2/L3 把“事实”和“判断”先做掉。它要把公告、财报、业绩预告、问询函回复、投资者交流记录等披露文件做成可以查询、可以精确引用、可以被 L2 直接抓取的数据：原始文件要保存，元数据要入库，正文、章节、表格、附件、定位和处理状态要入库，再加上结构标签、主题标签和检索索引，让 L2 抽取的每一条 claim 都能锚回原文。规范化口径、生成 claim、判真伪、判重要性、采信都留给 L2/L3。

理想使用方式是：我（或 L2 程序）想看某家公司某类公告、某份财报里的某段正文、某张表、某个章节或某类主题段落，可以直接从数据库/API 按公司、报告期、公告类型、章节、表名取到已做过基础处理、并能精确追回原文页码/表格/段落的数据，而不用重新打开 PDF。

## 参考形态

目标形态更接近 filings database，而不是文件管理器。

可以参考两类产品感觉：

- Financial Datasets 的 `filings/items`：按 ticker、filing type、year、quarter、item 查询 filing 里的分段正文和附件文本。
- 巨潮 CNINFO WebAPI 文档：每个接口都有 URL、输入参数、输出字段、结果格式、字段选择、条数限制和排序。

本项目不是要照抄它们的接口，而是学习这种数据产品形态：公告和财报进入系统后，应该变成结构化、可查询、可复用的本地数据。

再往后一层看，它也不是要把所有资料塞进向量库里问答。它是 L1 单层，但内部有一条“原文 → 可寻址锚”的派生梯度，整条梯度都停在 L1，不跨进事实与判断：

- raw 段：原始披露文件、来源、版本、哈希、parser run 都能复查、可重跑；
- 锚段（仍属 L1）：解析出的章节 / item / 段落 / 表格 / 单元格，都有稳定 id + 精确定位（页码 / bbox / 段落序号），再加结构标签、主题标签和检索索引，让 L2 好抓、好发现重点；
- 边界：规范化成“指标事实”、生成 claim、冲突检测与对账、判真伪、采信，都不在这里做——那是 L2/L3 的活。检索（关键词、全文、embedding）只作候选发现，不是事实来源。

## 这个服务做什么

当前范围只包括上市公司披露文件：

- 年报、半年报、季报；
- 业绩预告、业绩快报；
- 投资者关系活动记录表、业绩说明会记录、投资者交流 / 调研活动记录；
- 普通公告、重大事项公告；
- 问询函、监管函及回复；
- 其他需要保留原文的交易所披露文件。

第一阶段可以先不追求全自动全量覆盖。可以由人先挑一些重要公司、重要公告或重要财报，让服务先把这些资料处理好。

服务要做的事情：

1. 获取或导入公告 / 财报 PDF。
2. 保存原始 PDF 到本地硬盘。
3. 记录来源、下载时间、公告日期、标题、公司代码、公告编号、文件哈希、存储路径。
4. 做基础去重，避免同一公告反复入库。
5. 解析 PDF 文本。
6. 解析表格，保留页码和表格位置。
7. 尽量识别公告类型、公司、日期、报告期、市场、证券代码等基础字段。
8. 将正文切成可查询的段落、章节或 item。
9. 将表格整理成可查询的表格记录，保留表名、页码、行列和原始文本。
10. 把原文、元数据、解析结果、处理版本和处理日志写入本地数据库。

## 这个服务不做什么

这个服务不负责全部投研资料。

以下内容不属于它的核心 raw 归档范围：

- 采购的数据库数据；
- 券商、Wind、iFinD、Choice 等 API 数据；
- web search 结果；
- 非交易所披露的人工上传纪要、笔记、PPT、Excel、图片、录音转写；
- 研报、新闻、自媒体、论坛、社群内容。

这些资料以后可以由别的服务、API 或人工流程进入系统。`disclosure_anchor` 只把公告和财报这部分 raw data 做扎实。

Wind、Tushare、iFinD、Choice 等外部结构化数据**不在本服务范围内**。按 v0.5 §2.3，L1 对这类 Tier 1 外部数据只登记 `source_ref`（provider、dataset、query、as-of time、口径、来源哈希），用来说明“曾经看到过什么、何时、以什么形式”，但**不在这里把它规范化成“指标事实”，也不和披露字段做对账**——那是 L2 的事。本服务只保证：将来 L2 要把外部指标和披露表格并列、对账时，能从这里精确锚回披露侧的原文。

它也不做：

- 估值、选股、买卖建议；
- 净利润预测；
- claim 最终采信；
- 完整证据账本；
- 完整研究知识图谱和图推理系统；
- 自动研究结论生成。

换句话说，`disclosure_anchor` 只保证披露侧的每个单元都可被精确寻址、可被锚回（让 L2 的 evidence link 有稳定落点），但它不生成 claim、不建证据账本、不做 claim 仲裁、不做字段关系的真相裁定和预测推理。

## 输入

更准确地说，这个服务的输入不是“人给几个 PDF”本身，而是 **数据源记录、覆盖范围和处理任务**。

核心输入应该分成几类：

- 覆盖范围：公司 / 证券代码、市场、公告类别、日期区间、报告期、优先级；
- 来源配置：CNINFO、交易所、港交所等披露渠道的接口、查询参数、分类字段和文件 URL 规则；
- `source_record`：从披露渠道拿到的公告索引记录，包括公告 ID、标题、公告日期、公司代码、公告类别、PDF/H5/附件 URL、远端原始 payload、获取时间；
- `raw_file`：由 `source_record` 指向并下载 / 保存的 PDF、H5 保存件、附件；
- 处理任务：哪些 `source_record` 或 `raw_file` 需要下载、解析、重跑、抽表、抽关键文字；
- 外部结构化数据访问记录：Wind、Tushare、iFinD、Choice 等 provider 的 query、dataset、as-of time、返回摘要和口径，供 L2 后续对账或补充字段时引用。

人工输入更适合作为 **范围和优先级**：

- 人工指定一批公司；
- 人工指定一批重要公告 ID、公告链接或 PDF；
- 人工把某些公告标成“优先解析”或“优先作为 L2 候选锚”。

第一版可以先由人工给公司、公告链接或本地 PDF 作为 seed，但入库时仍然应该转成统一的 `source_record`、`raw_file`、`processing_run`，不要把系统设计成只靠手工文件导入的工具。

## 输出

服务输出的是一个可查询的本地公告 / 财报数据库。它只有**一类输出：可寻址、可发现重点的 L1 披露锚**，不产出“事实”或“判断”。这层锚里有一条“原文 → 解析 → 可寻址锚”的派生梯度，但整条梯度都关在 L1。

raw / parsed 部分：

- 原始 PDF 文件；
- 公告 / 财报元数据记录；
- 文件哈希和存储路径；
- PDF 文本抽取结果；
- PDF 表格抽取结果；
- 章节、段落或 item 级正文记录；
- 附件记录和附件文本；
- 页码、段落、表格、单元格定位信息；
- 处理状态、parser 版本和错误日志。

可寻址 / 可发现部分（仍是 L1，机械标注、不下结论）：

- 稳定单元 id：每个 section / item / block / table / cell 都能被独立、精确引用；
- 结构标签：标出这是“三大报表 / 管理层讨论 / 分产品收入表 / 投关问答”等；
- 主题标签：产能 / 价格 / 订单 / 客户 / 风险 等机械主题分类；
- 检索索引：全文 + 关键词 +（可选）embedding，只作候选发现，不作事实。

这层输出的交付对象是 **L2，不是 L3**。它不直接等于研究结论，也不是 L3 能直接采信的事实；它是让 L2 极容易产出事实的底座。明确**不在**本服务输出里（属 L2/L3）：规范化口径 / 单位 / 会计基础、`metric_observation` 当事实、生成 `claim`、`key_text` 当判断、冲突检测与对账、判真伪、判“对预测是否重要”、采信。

## 数据库里应该有什么

先不写具体 schema。L1 自维护的对象（本服务的核心）至少应该表达：

- company：公司、证券代码、市场、简称等基础信息；
- filing：一份公告或财报的主记录，包括标题、公告日期、报告期、公告类型、来源 URL、文件哈希、存储路径；
- filing_file：原始 PDF、附件、H5 页面等文件记录；
- filing_text：全文、章节、段落、item 的文本记录，带稳定 id 和定位；
- filing_table：从 PDF 中抽出来的表格记录，保留页码、bbox、原始结构；
- filing_table_cell：表格单元格、行列位置、原始值和定位（保留结构，不做口径规范化）；
- processing_run：每次下载、解析、重跑的 parser 版本、状态、错误和耗时；
- structural_tag / topic_tag：对单元的机械结构标签和主题标签，用于发现重点；
- query_index：全文 / 关键词 /（可选）embedding 等候选发现索引；
- source_ref：本服务自身及外部来源（含 Wind/Tushare 等 provider）的访问登记，只记“来自哪里、何时、以什么形式”，不存规范化事实。

明确属于 L2/L3、本服务**不产出**的对象（这里只为它们留下可精确锚回的定位，不替它们建库）：

- `metric_observation`（把数字规范化成指标事实）；
- `key_text` 当判断、`claim_candidate`（候选判断 / 候选事实）；
- 冲突检测、对账、采信，以及跨证据 `relationship`、`evidence_link` 的真相裁定侧。

数据库的目标是让 L2 程序直接按公司 / 报告期 / 公告类型 / 章节 / 表名问到锚，而不是每次重新打开 PDF，也不用自己再切一遍 PDF。

## 最小闭环

第一阶段的最小闭环是：

```text
给定覆盖范围 / 人工 seed / source_record
→ 发现或导入公告索引记录
→ 获取 / 导入公告 PDF
→ 保存原始文件
→ 记录元数据和哈希
→ 抽取文本和表格
→ 写入本地公告数据库
→ 能按公司、日期、公告类型、报告期查询
→ 能直接读到基础处理后的正文和表格
→ 能给每个单元打结构 / 主题标签并建检索索引
→ 能按公司 / 报告期 / 公告类型 / 章节 / 表名把锚交给 L2
```

只要这个闭环稳定，就可以继续扩展自动抓取、更多公告类型、更好的表格解析和更细的发现标签——但抽事实、生成 claim 仍然留给 L2。

## 判断是否做对

一个可用的第一版应该能回答：

- 某家公司有哪些公告和财报已经入库？
- 某份公告的原始 PDF 存在哪里？
- 这份文件是不是重复下载过？
- 它的公告日期、标题、公司代码、公告类型是什么？
- 它对应哪个报告期？
- 文本和表格有没有解析成功？
- 某个章节、段落或 item 的正文能不能直接查出来？
- 投资者交流记录里的问答、参与机构、调研时间等内容能不能直接查出来？
- 某个章节、表格或主题段落能不能被精确定位到哪份公告、哪一页、哪个表、哪段原文，让 L2 抽出的 claim 锚得回去？
- Wind/Tushare 等外部来源的访问有没有被登记成 `source_ref`（来源、query、as-of time、口径），方便 L2 以后对账？
- L2 要为某个报告期净利润找证据时，能不能按公司 / 报告期 / 公告类型 / 章节 / 表名快速捞到候选锚，而不用重读全文？
- 某个表格来自 PDF 的哪一页？
- 某张表能不能以表格记录形式直接读出来？
- 哪些文件处理失败，为什么失败？

如果这些问题能稳定回答，这个服务的目的就达到了：把公告和财报 raw data 变成可保存、可查询、可复查、**可被 L2 精确寻址和发现重点**的本地披露锚库——它不替 L2/L3 下结论，但让它们的活变得很轻。
