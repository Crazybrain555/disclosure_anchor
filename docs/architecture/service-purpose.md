# disclosure_anchor 服务目的

参考协议：`docs/reference/投研预测引擎顶层框架协议_v0.5.md`

## 一句话

`disclosure_anchor` 是一个本地上市公司公告 / 财报数据库服务。

它不是只把 PDF 下载到文件夹里，而是把公告、财报、业绩预告、问询函回复等披露文件做成可以查询的数据：原始文件要保存，元数据要入库，正文、章节、表格、附件和处理状态也要入库。理想使用方式是：我想看某家公司某类公告、某份财报里的某段正文或某张表，可以直接从数据库/API 拿到已经做过基础处理的数据。

## 参考形态

目标形态更接近 filings database，而不是文件管理器。

可以参考两类产品感觉：

- Financial Datasets 的 `filings/items`：按 ticker、filing type、year、quarter、item 查询 filing 里的分段正文和附件文本。
- 巨潮 CNINFO WebAPI 文档：每个接口都有 URL、输入参数、输出字段、结果格式、字段选择、条数限制和排序。

本项目不是要照抄它们的接口，而是学习这种数据产品形态：公告和财报进入系统后，应该变成结构化、可查询、可复用的本地数据。

## 这个服务做什么

当前范围只包括上市公司披露文件：

- 年报、半年报、季报；
- 业绩预告、业绩快报；
- 普通公告、重大事项公告；
- 问询函、监管函及回复；
- 其他需要保留原文的交易所披露文件。

第一阶段可以先不追求全自动全量覆盖。可以由人先挑一些重要公司、重要公告或重要财报，让服务先把这些资料处理好。

服务要做的事情：

1. 获取或导入公告 / 财报 PDF。
2. 保存原始 PDF 到本地硬盘。
3. 记录来源、下载时间、公告日期、标题、公司代码、公告编号、文件哈希、存储路径。
4. 做基础去重，避免同一公告反复入库。
5. 解析 PDF 文本。
6. 解析表格，保留页码和表格位置。
7. 尽量识别公告类型、公司、日期、报告期、市场、证券代码等基础字段。
8. 将正文切成可查询的段落、章节或 item。
9. 将表格整理成可查询的表格记录，保留表名、页码、行列和原始文本。
10. 把原文、元数据、解析结果、处理版本和处理日志写入本地数据库。

## 这个服务不做什么

这个服务不负责全部投研资料。

以下内容不属于它的核心范围：

- 采购的数据库数据；
- 券商、Wind、iFinD、Choice 等 API 数据；
- web search 结果；
- 人工上传的纪要、笔记、PPT、Excel、图片、录音转写；
- 研报、新闻、自媒体、论坛、社群内容。

这些资料以后可以由别的服务、API 或人工流程进入系统。`disclosure_anchor` 只把公告和财报这部分 raw data 做扎实。

它也不做：

- 估值、选股、买卖建议；
- 净利润预测；
- claim 最终采信；
- 完整证据账本；
- 复杂知识图谱；
- 自动研究结论生成。

## 输入

输入可以很简单：

- 人工给定的一批公司代码；
- 人工挑选的一批公告链接或 PDF；
- 从巨潮资讯、交易所、港交所等披露渠道发现的公告；
- 后续自动任务发现的新公告。

第一版可以先支持人工挑选的重要公告。先把处理链路跑通，比一开始做全市场全量覆盖更重要。

## 输出

服务输出的是一个可查询的本地公告 / 财报数据库：

- 原始 PDF 文件；
- 公告 / 财报元数据记录；
- 文件哈希和存储路径；
- PDF 文本抽取结果；
- PDF 表格抽取结果；
- 章节、段落或 item 级正文记录；
- 附件记录和附件文本；
- 页码、段落、表格定位信息；
- 处理状态和错误日志；
- 可供后续程序查询的本地数据库记录或本地 API。

这些输出不直接等于研究结论。它们是后续研究、证据抽取、预测模型使用的底层材料。

## 数据库里应该有什么

先不写具体 schema，但数据库至少应该表达几类对象：

- company：公司、证券代码、市场、简称等基础信息；
- filing：一份公告或财报的主记录，包括标题、公告日期、报告期、公告类型、来源 URL、文件哈希、存储路径；
- filing_file：原始 PDF、附件、H5 页面等文件记录；
- filing_text：全文、章节、段落、item 的文本记录；
- filing_table：从 PDF 中抽出来的表格记录；
- filing_table_cell：需要细化时保存表格单元格、行列位置和原始值；
- processing_run：每次下载、解析、重跑的版本、状态、错误和耗时；
- query_index：后续为了快速检索而建立的索引或派生字段。

数据库的目标是让程序直接问数据，而不是每次重新打开 PDF。

## 最小闭环

第一阶段的最小闭环是：

```text
人工挑选公司或公告
→ 获取 / 导入公告 PDF
→ 保存原始文件
→ 记录元数据和哈希
→ 抽取文本和表格
→ 写入本地公告数据库
→ 能按公司、日期、公告类型、报告期查询
→ 能直接读到基础处理后的正文和表格
```

只要这个闭环稳定，就可以继续扩展自动抓取、更多公告类型、更好的表格解析和后续证据抽取。

## 判断是否做对

一个可用的第一版应该能回答：

- 某家公司有哪些公告和财报已经入库？
- 某份公告的原始 PDF 存在哪里？
- 这份文件是不是重复下载过？
- 它的公告日期、标题、公司代码、公告类型是什么？
- 它对应哪个报告期？
- 文本和表格有没有解析成功？
- 某个章节、段落或 item 的正文能不能直接查出来？
- 某个表格来自 PDF 的哪一页？
- 某张表能不能以表格记录形式直接读出来？
- 哪些文件处理失败，为什么失败？

如果这些问题能稳定回答，这个服务的目的就达到了：把公告和财报 raw data 变成可保存、可查询、可复查、可被程序直接使用的本地数据库。
